## FP-006 — Gestione task preset

---

### 1. Obiettivo funzionale

La funzionalità “Gestione task preset” consente all’utente di configurare e utilizzare tipologie predefinite di attività AI, chiamate **task preset**, per stimare in modo più realistico il consumo di Microsoft Copilot Cowork Credits.

Un task preset rappresenta una categoria ricorrente di utilizzo, ad esempio:

* simple prompt;
* document summarization;
* document analysis;
* deep research;
* agent workflow;
* multi-step workflow;
* customer support automation;
* sales research;
* legal document review;
* HR policy analysis;
* report generation;
* browser-based research;
* image generation.

L’obiettivo è permettere all’utente di costruire una stima non solo per profilo generico di utilizzo, ma anche per tipo di attività effettivamente svolta.

I task preset devono aiutare l’utente a rispondere a domande come:

* Quali attività consumano più crediti?
* Quanto pesa la deep research rispetto a un prompt semplice?
* Quanto incide un workflow multi-step?
* Quali task sono più adatti a un segmento specifico?
* Quali task sono light, medium o heavy?
* Quali task usano browser, tool, contesto esteso o runtime elevato?
* Quali task potrebbero richiedere modelli premium?
* Come cambia il costo se un segmento fa più legal review o più report generation?

I task preset rendono quindi l’app più vicina a un vero cost calculator consulenziale.

---

### 2. Attori coinvolti

Gli attori principali sono:

* Consulente;
* Solution architect;
* Sales specialist;
* Account manager;
* Responsabile IT;
* Responsabile Finance;
* Responsabile Legal;
* Responsabile HR;
* Responsabile Operations;
* Cliente finale durante workshop;
* Utente avanzato che personalizza le assunzioni.

---

### 3. Prerequisiti

Per usare i task preset devono essere disponibili:

* assumption pack di default;
* credit bands light/medium/heavy;
* profili di utilizzo;
* modelli AI;
* segmenti workforce;
* motore di calcolo;
* servizio di validazione;
* servizio di salvataggio locale;
* eventuale wizard scenario.

La funzionalità può essere usata in tre modalità:

1. **Modalità catalogo**

   * l’utente visualizza e gestisce tutti i task preset disponibili.

2. **Modalità scenario**

   * l’utente assegna un mix di task a uno scenario o a un segmento.

3. **Modalità advanced calculation**

   * l’utente costruisce un oggetto custom di consumo per simulazioni dettagliate.

---

### 4. Trigger

La funzionalità viene attivata quando:

* l’utente apre la pagina “Task presets”;
* l’utente entra nello step “Usage profiles & task mix” del wizard;
* l’utente modifica il task mix di un segmento;
* l’utente crea un task custom;
* l’utente duplica un task preset standard;
* l’utente importa un JSON contenente task custom;
* l’utente apre uno scenario condiviso con task preset custom;
* l’utente seleziona un task nel calculator;
* il sistema rileva un task preset mancante in uno scenario importato.

---

### 5. Descrizione funzionale dettagliata

Il sistema deve fornire un catalogo iniziale di task preset di sistema.

Ogni task preset deve rappresentare un tipo di attività Cowork e contenere:

* nome;
* descrizione;
* classe di intensità;
* crediti stimati min/mid/max;
* modello consigliato;
* complessità del contesto;
* uso di tool;
* runtime previsto;
* uso browser;
* uso immagini;
* esempi concreti;
* segmenti consigliati;
* note e disclaimer.

I task preset standard devono essere sempre disponibili, non eliminabili e non modificabili direttamente.

L’utente deve poter:

* visualizzare task preset standard;
* duplicare task preset standard;
* creare task preset custom;
* modificare task preset custom;
* eliminare task preset custom non utilizzati;
* associare task preset a usage profile;
* associare task preset a segmenti;
* definire un task mix custom per segmento;
* esportare task preset custom;
* importare task preset custom;
* vedere l’impatto di un task sul consumo stimato.

Il task preset non deve essere presentato come una tariffa ufficiale Microsoft. Deve essere mostrato come oggetto di stima basato su assumption pack.

### 8. Regole di business

#### BR-FP006-001 — Task preset standard sempre disponibili

I task preset standard devono essere sempre disponibili dopo inizializzazione dell’app.

Non possono essere eliminati.

---

#### BR-FP006-002 — Task preset standard non modificabili direttamente

I preset di sistema non devono essere modificabili direttamente.

L’utente può duplicarli e modificare la copia.

---

#### BR-FP006-003 — Nome task obbligatorio

Ogni task preset deve avere un nome.

Non è possibile salvare un task custom senza nome.

---

#### BR-FP006-004 — Intensità obbligatoria

Ogni task preset deve avere una classe di intensità:

* light;
* medium;
* heavy;
* custom.

La classe di intensità determina la banda base di consumo.

---

#### BR-FP006-005 — Crediti non negativi

I valori:

* `creditsMin`;
* `creditsMid`;
* `creditsMax`;

devono essere maggiori o uguali a zero.

Valori negativi sono bloccanti.

---

#### BR-FP006-006 — Coerenza min/mid/max

Per i task range-based deve valere:

```text id="eu49l2"
creditsMin <= creditsMid <= creditsMax
```

Se la condizione non è rispettata, il sistema deve mostrare errore bloccante.

---

#### BR-FP006-007 — Heavy con massimo configurabile

Poiché alcune bande heavy possono essere pubblicate come soglia minima o valore aperto, l’app deve richiedere un valore massimo configurabile nell’assumption pack o nel task custom.

Se `creditsMax` non è definito, il sistema deve usare il default dell’assumption pack e mostrare warning.

---

#### BR-FP006-008 — Modello default obbligatorio

Ogni task preset deve avere un modello default.

Se l’utente non seleziona un modello, il sistema deve usare `Auto`.

---

#### BR-FP006-009 — Modelli consigliati

Un task può avere uno o più modelli consigliati.

Esempio:

* Simple prompt → Auto / Sonnet;
* Deep research → Auto / Opus / GPT;
* Legal review → Opus / GPT;
* Image generation → Imagen.

Questi suggerimenti non devono essere vincolanti, salvo task incompatibili.

---

#### BR-FP006-010 — Task image-related

Se `imageUsage` è diverso da `none`, il sistema deve mostrare warning sul potenziale consumo aggiuntivo.

Il task dovrebbe suggerire modelli compatibili con immagini.

---

#### BR-FP006-011 — Task browser-related

Se `browserUsage` è diverso da `none`, il sistema deve mostrare warning sul potenziale consumo aggiuntivo legato a browser/runtime.

---

#### BR-FP006-012 — Task tool-heavy

Se `toolsUsage = high`, il sistema deve evidenziare che il task può consumare più crediti per chiamate a tool, skill o workflow.

---

#### BR-FP006-013 — Task custom

L’utente può creare task preset custom.

I task custom devono essere:

* modificabili;
* eliminabili se non in uso;
* esportabili;
* importabili;
* marcati come custom nei report.

---

#### BR-FP006-014 — Task in uso

Se un task custom è usato da uno o più scenari, segmenti o profili, non deve essere eliminato senza gestione dipendenze.

Per MVP: impedire eliminazione finché il task è in uso.

---

#### BR-FP006-015 — Override a livello segmento

Un segmento può usare un task preset, ma può anche sovrascrivere quantità o fattori.

Gli override non devono modificare il task preset originale.

---

#### BR-FP006-016 — Task mix per profilo

Un usage profile può essere costruito anche come mix di task preset.

Esempio:

* Medium Sales profile:

  * 20 simple prompt;
  * 8 sales research;
  * 4 report generation;
  * 2 deep research.

Questa funzionalità può essere avanzata e facoltativa nel primo MVP.

---

#### BR-FP006-017 — Task mix per segmento

Un segmento può avere un task mix personalizzato basato su preset.

Esempio:

* Legal team:

  * 10 document analysis;
  * 5 legal review;
  * 2 deep research.

Il task mix custom prevale sui valori generici light/medium/heavy del usage profile.

---

#### BR-FP006-018 — Task preset importato

Un task importato da JSON deve essere validato.

Se fa riferimento a modelli o assumption pack mancanti, il sistema deve tentare mapping o mostrare warning.

---

#### BR-FP006-019 — Task preset e assumption pack

Ogni task preset deve indicare se i valori di credito derivano:

* da assumption pack di sistema;
* da assumption pack custom;
* da valore custom del task;
* da import.

---

#### BR-FP006-020 — Disclaimer

La UI deve chiarire che i valori di consumo dei task preset sono assunzioni operative e non una rate card ufficiale.

---

### 9. Logiche di calcolo

Il task preset contribuisce al calcolo definendo il costo stimato per unità di task.

In modalità range:

```text id="mmhgk5"
taskCreditsMin = creditsMin
taskCreditsMid = creditsMid
taskCreditsMax = creditsMax
```

Applicazione fattori del task:

```text id="emltev"
adjustedTaskCreditsMin =
  taskCreditsMin
  × modelFactor
  × contextFactor
  × toolsFactor
  × runtimeFactor
  × browserFactor
  × imageFactor

adjustedTaskCreditsMid =
  taskCreditsMid
  × modelFactor
  × contextFactor
  × toolsFactor
  × runtimeFactor
  × browserFactor
  × imageFactor

adjustedTaskCreditsMax =
  taskCreditsMax
  × modelFactor
  × contextFactor
  × toolsFactor
  × runtimeFactor
  × browserFactor
  × imageFactor
```

Se il segmento usa un task mix:

```text id="1tw6sh"
segmentTaskCreditsMin =
  sum(task.quantityPerActiveUserPerMonth × activeUsers × adjustedTaskCreditsMin)

segmentTaskCreditsMid =
  sum(task.quantityPerActiveUserPerMonth × activeUsers × adjustedTaskCreditsMid)

segmentTaskCreditsMax =
  sum(task.quantityPerActiveUserPerMonth × activeUsers × adjustedTaskCreditsMax)
```

Se il profilo usa task preset:

```text id="4pfnlm"
profileMonthlyCreditsPerUser =
  sum(task.quantityPerUserPerMonth × adjustedTaskCredits)
```

Se un task preset è mappato solo a intensità light/medium/heavy:

```text id="pg2sdy"
taskCredits =
  assumptionPack.creditBands[task.intensity]
```

Priorità dei valori effettivi:

```text id="i4sjkf"
effectiveTaskCredits =
  segmentTaskOverride
  ?? customTaskPresetCredits
  ?? assumptionPack.taskPresetCredits
  ?? assumptionPack.creditBandByIntensity
```

### 10. Flusso principale — Visualizzazione catalogo task preset

1. L’utente apre la sezione “Task presets”.
2. Il sistema carica task preset standard e custom.
3. Il sistema mostra una lista o griglia di task.
4. Ogni task mostra:

   * nome;
   * intensità;
   * crediti min/mid/max;
   * modello default;
   * driver principali;
   * stato system/custom;
   * warning.
5. L’utente può filtrare per:

   * intensità;
   * modello;
   * categoria;
   * system/custom;
   * browser usage;
   * image usage.
6. L’utente può aprire il dettaglio di un task.
7. Se il task è standard, può duplicarlo.
8. Se il task è custom, può modificarlo o eliminarlo.

---

### 11. Flusso principale — Creazione task custom

1. L’utente clicca “Create task preset”.
2. Il sistema apre il form.
3. L’utente inserisce:

   * nome;
   * descrizione;
   * intensità;
   * crediti min/mid/max;
   * modello default;
   * fattori context/tool/runtime/browser/image;
   * esempi;
   * note.
4. Il sistema valida i campi.
5. L’utente salva.
6. Il sistema crea il task custom con nuovo ID.
7. Il task viene salvato in localStorage.
8. Il task diventa disponibile nei task mix di profili e segmenti.

---

### 12. Flusso principale — Uso task preset nel segmento

1. L’utente apre un segmento in modalità avanzata.
2. Seleziona “Use custom task mix”.
3. Clicca “Add task”.
4. Seleziona un task preset dal catalogo.
5. Inserisce quantità mensile per utente attivo.
6. Il sistema calcola l’impatto del task sul segmento.
7. L’utente aggiunge altri task.
8. Il sistema aggiorna il totale segmento.
9. Il sistema aggiorna il totale scenario.

---

### 13. Flussi alternativi

#### A1 — Duplicazione task preset standard

1. L’utente apre un task standard.
2. Clicca “Duplicate”.
3. Il sistema crea una copia custom.
4. Il nome diventa “Copy of [task name]”.
5. L’utente modifica i valori.
6. Il sistema salva il task custom.

---

#### A2 — Modifica task custom

1. L’utente apre un task custom.
2. Modifica crediti, fattori o descrizione.
3. Il sistema valida i dati.
4. Se il task è in uso, mostra impatto sugli scenari collegati.
5. L’utente conferma.
6. Il sistema salva.
7. Gli scenari collegati vengono marcati da ricalcolare.

---

#### A3 — Eliminazione task custom non usato

1. L’utente apre un task custom non in uso.
2. Clicca “Delete”.
3. Il sistema chiede conferma.
4. L’utente conferma.
5. Il task viene eliminato.

---

#### A4 — Eliminazione task custom in uso

1. L’utente tenta di eliminare un task custom usato.
2. Il sistema mostra warning.
3. Per MVP, il sistema impedisce l’eliminazione.
4. L’utente deve prima rimuoverlo dai task mix.

---

#### A5 — Task con crediti incoerenti

1. L’utente inserisce `creditsMin` maggiore di `creditsMid` o `creditsMax`.
2. Il sistema mostra errore.
3. Il salvataggio è bloccato.

---

#### A6 — Task heavy senza max definito

1. L’utente crea task heavy.
2. Non valorizza `creditsMax`.
3. Il sistema usa default assumption pack, se disponibile.
4. Se non disponibile, blocca il salvataggio o richiede valore.
5. Mostra warning sulla natura stimata del massimo.

---

#### A7 — Task image generation

1. L’utente crea o seleziona un task con image usage.
2. Il sistema mostra warning su possibile consumo specifico.
3. Il sistema suggerisce modelli compatibili.
4. Il task viene marcato come image-related.

---

#### A8 — Task browser-heavy

1. L’utente crea o seleziona un task con browser usage heavy.
2. Il sistema mostra warning su browser/runtime.
3. Il task viene marcato come browser-heavy.
4. Il calcolo applica browser factor.

---

#### A9 — Import task preset

1. L’utente importa JSON.
2. Il payload contiene task preset custom.
3. Il sistema valida i task.
4. I task validi vengono importati.
5. I task con problemi vengono marcati “Needs review”.

---

#### A10 — Task importato con modello mancante

1. Il task importato fa riferimento a un modello non disponibile.
2. Il sistema assegna Auto.
3. Mostra warning.
4. Il task resta importabile.

---

#### A11 — Uso task preset in usage profile

1. L’utente crea un usage profile custom.
2. Seleziona modalità “Task preset mix”.
3. Aggiunge task e quantità.
4. Il sistema calcola automaticamente la distribuzione light/medium/heavy.
5. Il profilo viene salvato.

---

#### A12 — Rimozione task da task mix

1. L’utente apre un task mix segmento.
2. Rimuove un task.
3. Il sistema aggiorna il calcolo del segmento.
4. Se il task mix resta vuoto, mostra warning.

---

### 14. Validazioni

#### Validazioni bloccanti

* nome task mancante;
* nome composto solo da spazi;
* intensità mancante;
* crediti negativi;
* `creditsMin > creditsMid`;
* `creditsMid > creditsMax`;
* modello default mancante;
* fattori negativi;
* task standard modificato direttamente;
* task standard eliminato;
* eliminazione task custom in uso, per MVP;
* import task con schema non valido;
* quantità task per utente negativa in un task mix.

---

#### Warning non bloccanti

* task con crediti tutti a zero;
* task heavy con valore max molto alto;
* task con browser usage heavy;
* task con image usage;
* task con tools usage high;
* task con runtime veryLong;
* task con modello premium;
* task custom usato in molti scenari;
* task importato da schema vecchio;
* task senza descrizione;
* task senza esempi;
* task con nome duplicato;
* task non usato da nessuno scenario;
* task con factor molto elevati;
* task basato su assumption pack obsoleto.

### 20. Criteri di accettazione

#### AC-001 — Task preset standard disponibili

Given l’utente apre la pagina Task presets,
when l’app è inizializzata,
then il sistema mostra il catalogo standard di task preset.

---

#### AC-002 — Task standard non modificabile

Given l’utente apre un task standard,
when prova a modificarlo direttamente,
then il sistema impedisce la modifica e propone la duplicazione.

---

#### AC-003 — Task standard non eliminabile

Given l’utente apre un task standard,
when prova a eliminarlo,
then il sistema impedisce l’eliminazione.

---

#### AC-004 — Creazione task custom valido

Given l’utente apre il form nuovo task,
when inserisce nome, intensità, crediti validi e modello default,
then il sistema salva il task custom.

---

#### AC-005 — Nome obbligatorio

Given l’utente crea un task preset,
when lascia vuoto il nome,
then il sistema mostra errore bloccante.

---

#### AC-006 — Intensità obbligatoria

Given l’utente crea un task preset,
when non seleziona intensità,
then il sistema mostra errore bloccante.

---

#### AC-007 — Crediti negativi non validi

Given l’utente crea o modifica un task,
when inserisce crediti negativi,
then il sistema mostra errore bloccante.

---

#### AC-008 — Range coerente

Given l’utente crea un task range-based,
when `creditsMin <= creditsMid <= creditsMax`,
then il sistema considera il range valido.

---

#### AC-009 — Range incoerente

Given l’utente crea un task range-based,
when `creditsMin > creditsMid` oppure `creditsMid > creditsMax`,
then il sistema impedisce il salvataggio.

---

#### AC-010 — Duplicazione task standard

Given l’utente apre un task standard,
when clicca Duplicate,
then il sistema crea un task custom con nuovo ID e stessi valori.

---

#### AC-011 — Modifica task custom

Given esiste un task custom,
when l’utente modifica i valori e salva,
then il task viene aggiornato e `updatedAt` cambia.

---

#### AC-012 — Eliminazione task custom non usato

Given esiste un task custom non usato,
when l’utente conferma eliminazione,
then il task viene eliminato.

---

#### AC-013 — Task custom in uso non eliminabile

Given un task custom è usato in un task mix,
when l’utente tenta di eliminarlo,
then il sistema impedisce l’eliminazione e mostra warning.

---

#### AC-014 — Uso task in segmento

Given l’utente apre un segmento con task mix custom,
when aggiunge un task preset con quantità mensile,
then il sistema calcola l’impatto del task sul segmento.

---

#### AC-015 — Quantità negativa nel task mix

Given l’utente aggiunge un task al task mix,
when inserisce quantità negativa,
then il sistema mostra errore bloccante.

---

#### AC-016 — Task browser-heavy

Given un task ha browser usage heavy,
when viene usato in uno scenario,
then il sistema mostra warning e applica browser factor.

---

#### AC-017 — Task image-related

Given un task ha image usage diverso da none,
when viene usato in uno scenario,
then il sistema mostra warning e applica image factor.

---

#### AC-018 — Task tool-heavy

Given un task ha tools usage high,
when viene usato in uno scenario,
then il sistema mostra warning e applica tools factor.

---

#### AC-019 — Import task validi

Given l’utente importa un JSON con task preset validi,
when conferma import,
then i task vengono salvati e diventano disponibili nel catalogo.

---

#### AC-020 — Import task con modello mancante

Given un task importato fa riferimento a un modello non disponibile,
when il sistema valida il task,
then assegna modello Auto e mostra warning.

---

#### AC-021 — Export task custom

Given esistono task preset custom,
when l’utente esporta JSON,
then i task custom vengono inclusi nell’export.

---

#### AC-022 — Task impact preview

Given l’utente apre il dettaglio di un task,
when il task ha valori validi,
then il sistema mostra crediti e costo stimati per 1, 10 e 100 utenti attivi.

---

#### AC-023 — Breakdown per task

Given uno scenario usa task mix custom,
when il sistema calcola il risultato,
then il breakdown mostra il contributo di ogni task preset.

---

#### AC-024 — Task custom nel report

Given uno scenario usa un task custom,
when l’utente apre il report,
then il report indica che il task è custom.

---

#### AC-025 — Disclaimer assunzioni

Given l’utente visualizza valori di credito dei task preset,
when la UI mostra i numeri,
then deve essere visibile che si tratta di assunzioni di stima e non prezzi ufficiali vincolanti.

### 21. Priorità

Priorità funzionale: **Should have per MVP esteso**, **Must have per versione consulenziale avanzata**.

Il task preset non è strettamente necessario per una prima versione basata solo su profili Light/Medium/Heavy, ma è essenziale per rendere l’app un vero estimator consulenziale.

Con i task preset, l’app può:

* stimare attività reali;
* spiegare meglio i driver di consumo;
* differenziare task semplici e complessi;
* modellare workflow agentici;
* creare breakdown più credibili;
* supportare casi d’uso per industry;
* aiutare il cliente a capire da dove nasce il costo.

### 22. Note UX

La gestione task preset deve essere potente ma non troppo tecnica.

Principi UX:

* usare un catalogo visuale con card;
* mostrare intensità con badge;
* spiegare ogni task con esempi concreti;
* evidenziare task heavy, browser, image e tool-heavy;
* mostrare sempre min/mid/max;
* permettere duplicazione rapida;
* evitare modifica diretta dei preset standard;
* nascondere factor avanzati in sezione espandibile;
* mostrare impatto su 1, 10 e 100 utenti;
* rendere semplice l’aggiunta di task a un segmento;
* chiarire che i numeri sono stime;
* usare tooltip per context, tools, runtime, browser e image;
* dare suggerimenti di modello senza renderli vincolanti.
