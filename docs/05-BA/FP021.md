## FP-021 — Report scenario

---

### 1. Obiettivo funzionale

La funzionalità “Report scenario” consente all’utente di generare una vista finale, leggibile, stampabile ed esportabile di uno scenario di stima Microsoft Copilot Cowork Credits.

Il report deve consolidare in un unico documento:

* executive summary;
* dati company;
* configurazione scenario;
* risultati crediti;
* risultati economici;
* breakdown per segmento;
* breakdown per intensità;
* breakdown per task, se disponibile;
* breakdown per modello;
* funding plan;
* budget guardrails;
* assumption pack;
* formula di calcolo;
* warning;
* limitazioni;
* disclaimer;
* metadata;
* data del calcolo;
* stato dello scenario.

Il report deve servire sia per uso interno sia per condivisione con stakeholder esterni.

Casi d’uso principali:

* presentare la stima a un cliente;
* allegare una simulazione a una proposta;
* discutere uno scenario con Finance;
* confrontare alternative di rollout;
* documentare assunzioni usate;
* archiviare una stima storica;
* stampare o salvare in PDF tramite browser;
* copiare il contenuto in una presentazione;
* esportare dati di supporto in CSV o JSON.

Il report non deve presentare i valori come quotazione ufficiale o vincolante. Deve comunicare chiaramente che si tratta di una stima basata su assunzioni.

---

### 2. Attori coinvolti

Gli attori principali sono:

* Consulente;
* Solution architect;
* Sales specialist;
* Account manager;
* Partner Microsoft;
* Responsabile Finance;
* Responsabile Procurement;
* Responsabile IT;
* Responsabile AI adoption;
* Cliente finale;
* Utente che prepara una proposta;
* Utente che deve validare la stima;
* Utente che archivia scenari;
* Utente che confronta scenari;
* Utente che esporta dati per analisi esterna.

---

### 3. Prerequisiti

Per generare un report completo devono essere disponibili:

* company;
* scenario;
* segmenti workforce;
* calculation result;
* breakdown risultati;
* assumption snapshot;
* funding plan;
* budget result, se configurato;
* warning e validazioni;
* metadata scenario;
* data calcolo;
* stato risultato.

Il report può essere generato anche con dati parziali, ma deve indicare chiaramente le sezioni non disponibili.

Esempi:

* scenario non calcolato;
* funding mancante;
* budget non configurato;
* task breakdown non disponibile;
* calculation result stale;
* scenario invalid.

---

### 4. Trigger

La funzionalità viene attivata quando:

* l’utente apre uno scenario salvato;
* l’utente completa il wizard;
* l’utente clicca “View report”;
* l’utente clicca “Print / Save PDF”;
* l’utente clicca “Download summary”;
* l’utente esporta uno scenario;
* l’utente apre uno scenario da link condiviso;
* l’utente importa uno scenario JSON;
* l’utente vuole condividere una stima;
* l’utente apre la comparison e vuole vedere il dettaglio di uno scenario;
* l’utente apre portfolio e seleziona uno scenario;
* il sistema deve produrre un output governabile e auditabile.

---

### 5. Descrizione funzionale dettagliata

Il report scenario deve essere una vista strutturata e leggibile.

Deve essere accessibile da:

* scenario detail;
* wizard review;
* breakdown risultati;
* portfolio;
* comparison;
* shared scenario preview;
* dashboard.

Il report deve essere organizzato in sezioni progressive:

1. **Report header**

   * titolo, company, scenario, status, data.

2. **Executive summary**

   * KPI principali.

3. **Scenario configuration**

   * descrizione configurazione usata.

4. **Usage and adoption**

   * utenti, segmenti, adozione, attività.

5. **Credit estimate**

   * crediti mensili e annuali min/mid/max.

6. **Cost estimate**

   * costi mensili e annuali min/mid/max.

7. **Funding plan**

   * PAYG, prepaid, existing capacity, blended.

8. **Budget guardrails**

   * budget, status, overrun, risk.

9. **Breakdown**

   * segmenti, intensità, task, modelli.

10. **Assumptions**

* assumption pack, versione, source date, custom values.

11. **How this was calculated**

* formula e logica di calcolo.

12. **Warnings and limitations**

* errori, warning, caveat, stale status.

13. **Export and sharing metadata**

* source, createdAt, calculatedAt, exportedAt.

14. **Disclaimer**

* natura stimata e non ufficiale del report.

La UI deve consentire all’utente di:

* visualizzare report;
* stampare;
* salvare PDF tramite browser;
* esportare CSV;
* esportare JSON;
* copiare link condivisibile;
* duplicare scenario;
* aprire “How this was calculated”;
* tornare al wizard per modifiche.

### 7. Sezioni del report

## 7.1 Report header

La parte iniziale del report deve mostrare:

* nome report;
* nome company;
* nome scenario;
* stato scenario;
* stato calcolo;
* data creazione scenario;
* data ultimo aggiornamento;
* data calcolo;
* data generazione report;
* source scenario:

  * local;
  * imported;
  * shared;
  * demo.

Esempio:

```text id="nyrd3q"
Copilot Cowork Credits Estimate
Company: Acme S.p.A.
Scenario: Full rollout FY27
Status: Reviewed
Calculation: Valid
Generated: 2026-06-24
```

## 7.14 How this was calculated section

Mostra una sintesi del calcolo.

Formula principale:

```text id="1oxdqu"
monthlyCredits =
  sum over included segments of:
    activeUsers
    × monthlyTasksPerActiveUser
    × baseCreditsPerTask
    × modelFactor
    × contextFactor
    × toolsFactor
    × runtimeFactor
    × browserFactor
    × imageFactor
```

Formula funding PAYG:

```text id="f51zt0"
monthlyCost =
  billableCredits × pricePerCredit
```

Il report può includere una versione breve della formula e rimandare al pannello completo “How this was calculated”.

### 8. Regole di business

#### BR-FP021-001 — Report disponibile per ogni scenario

Ogni scenario deve poter aprire una vista report.

Se lo scenario non è completo, il report mostra lo stato incompleto.

---

#### BR-FP021-002 — Report completo solo con calculation result valido

Un report completo richiede calculation result valido o parziale.

Se il result manca, il report mostra input e messaggio di risultato non disponibile.

---

#### BR-FP021-003 — Nessun ricalcolo indipendente

Il report non deve ricalcolare i valori in modo autonomo.

Deve usare calculation result, breakdown, funding result e trace generati dai servizi centrali.

---

#### BR-FP021-004 — Midpoint come default

Il report deve mostrare il midpoint come valore principale.

Min e max devono essere visibili come range.

---

#### BR-FP021-005 — Range sempre visibile

Il report deve evitare di mostrare un solo costo come se fosse “il costo”.

Deve presentare almeno:

* min;
* mid;
* max;
* oppure indicare chiaramente che si tratta di point estimate.

---

#### BR-FP021-006 — Assumption pack obbligatorio

Il report deve includere assumption pack, versione e source date.

Se mancano, il report deve mostrare warning forte.

---

#### BR-FP021-007 — Snapshot storico

Il report deve usare lo snapshot delle assunzioni usato al momento del calcolo, non il pack corrente se è cambiato.

---

#### BR-FP021-008 — Custom assumptions evidenziate

Ogni assumption custom deve essere evidenziata nel report.

---

#### BR-FP021-009 — Funding separato dal consumo

Il report deve separare chiaramente:

* consumo crediti;
* valorizzazione economica;
* budget.

---

#### BR-FP021-010 — Costi opzionali

Se funding plan manca o è incompleto, il report deve mostrare crediti ma non costi.

---

#### BR-FP021-011 — Budget opzionale

Se budget guardrail non è configurato, il report deve indicarlo senza generare errore.

---

#### BR-FP021-012 — Warning sempre inclusi

Warning rilevanti devono essere inclusi nel report.

Non devono essere nascosti solo nella UI interattiva.

---

#### BR-FP021-013 — Stale result evidenziato

Se il calculation result è stale, il report deve mostrarlo chiaramente in header e warning section.

---

#### BR-FP021-014 — Scenario invalid

Uno scenario invalid non deve produrre un report finale con numeri presentati come validi.

Può produrre un report di input e issue.

---

#### BR-FP021-015 — Print-friendly

Il report deve essere ottimizzato per stampa e salvataggio PDF da browser.

---

#### BR-FP021-016 — Export coerente

I valori nel report devono essere coerenti con:

* dashboard;
* breakdown;
* CSV export;
* JSON export;
* comparison.

---

#### BR-FP021-017 — Report da shared scenario

Se lo scenario arriva da link condiviso, il report deve indicare source shared e se non è salvato localmente.

---

#### BR-FP021-018 — Report da imported scenario

Se lo scenario è importato, il report deve indicare source imported e needs review se applicabile.

---

#### BR-FP021-019 — Dati demo

Se il report usa dati demo, deve essere chiaramente indicato.

---

#### BR-FP021-020 — Nessuna modifica dati

Aprire, stampare o esportare il report non deve modificare scenario o calculation result.

Può aggiornare solo metadata di export, se previsto.

---

#### BR-FP021-021 — Branding configurabile

Per versione avanzata, il report può supportare logo, footer e note personalizzate.

Per MVP basta un layout neutro.

---

#### BR-FP021-022 — Lingua report

Per MVP il report può essere nella lingua dell’app.

Per versione avanzata può supportare lingua selezionabile.

---

#### BR-FP021-023 — Report sintetico e completo

Il sistema può supportare due modalità:

1. **Summary report**

   * executive summary, costi, warning, disclaimer.

2. **Full report**

   * include breakdown, assumptions e formula.

Per MVP si consiglia full report con sezioni collapsible nella UI e print-friendly espanso.

---

#### BR-FP021-024 — Print non include controlli UI

La versione stampata/PDF non deve includere pulsanti, menu o controlli interattivi.

---

#### BR-FP021-025 — Disclaimer obbligatorio

Ogni versione del report deve includere disclaimer.

---

### 9. Logiche di costruzione report

## 9.1 View model report

```text id="b8zhg4"
buildScenarioReportViewModel(scenarioId):
  scenario = getScenario(scenarioId)
  company = getCompany(scenario.companyId)
  result = getCalculationResult(scenarioId)
  breakdown = getBreakdown(result)
  funding = getFundingResult(scenarioId)
  budget = getBudgetResult(scenarioId)
  assumptions = getAssumptionSnapshot(result)
  warnings = getScenarioWarnings(scenarioId)

  return ReportViewModel
```

---

## 9.2 Stato report

```text id="oxs2r6"
if scenario has blocking errors:
  reportStatus = "invalid"

else if calculationResult missing:
  reportStatus = "notCalculated"

else if calculationResult stale:
  reportStatus = "stale"

else if criticalWarnings exist:
  reportStatus = "needsReview"

else:
  reportStatus = "ready"
```

---

## 9.3 Valori range

```text id="wmw6iy"
displayRange =
  {
    min: valueMin,
    mid: valueMid,
    max: valueMax
  }
```

costPerActiveUser =
  monthlyCostMid / activeUsers

```text id="uen2uy"
costPerActiveUser = null
```

### 10. Flusso principale — Apertura report da scenario

1. L’utente apre uno scenario salvato.
2. Clicca “View report”.
3. Il sistema recupera dati scenario.
4. Recupera company.
5. Recupera calculation result.
6. Recupera breakdown.
7. Recupera funding.
8. Recupera budget.
9. Recupera assumptions e warning.
10. Costruisce report view model.
11. Mostra report.
12. L’utente può stampare, esportare, condividere o modificare scenario.

---

### 11. Flusso principale — Report da wizard review

1. L’utente completa il wizard.
2. Il sistema calcola scenario.
3. L’utente arriva alla review.
4. Clicca “Generate report”.
5. Il sistema mostra report preliminare.
6. L’utente può:

   * salvare scenario;
   * esportare CSV;
   * esportare JSON;
   * stampare;
   * tornare a modificare.

---

### 12. Flusso principale — Print / Save PDF

1. L’utente apre il report.
2. Clicca “Print / Save PDF”.
3. Il sistema attiva layout print.
4. Espande sezioni rilevanti.
5. Nasconde controlli interattivi.
6. Apre dialog stampa del browser.
7. L’utente stampa o salva PDF.
8. Il sistema torna a layout normale.

---

### 13. Flusso principale — Report da shared scenario

1. L’utente apre scenario da link condiviso.
2. Visualizza preview.
3. Clicca “View report”.
4. Il sistema genera report usando payload condiviso.
5. Mostra source shared.
6. Indica se lo scenario non è salvato localmente.
7. L’utente può salvarlo, esportarlo JSON o stampare.

---

### 14. Flussi alternativi

#### A1 — Scenario non calcolato

1. L’utente apre report.
2. Calculation result manca.
3. Il report mostra:

   * dati input;
   * warning “not calculated”;
   * CTA Calculate scenario.
4. Non mostra costi o crediti finali.

---

#### A2 — Scenario stale

1. Calculation result è stale.
2. Il report mostra banner in alto.
3. I valori sono marcati come stale.
4. L’utente può ricalcolare prima di condividere.

---

#### A3 — Scenario invalid

1. Scenario contiene errori bloccanti.
2. Report status = invalid.
3. Il report mostra errori e input.
4. Non mostra executive summary economico come valido.

---

#### A4 — Funding mancante

1. Crediti disponibili.
2. Funding mancante.
3. Report mostra credit estimate.
4. Cost estimate mostra N/A.
5. Funding section mostra CTA o warning.

---

#### A5 — Budget non configurato

1. Budget guardrail mancante.
2. Report mostra “Not configured”.
3. Nessun errore.
4. Può suggerire aggiunta budget.

---

#### A6 — Task breakdown non disponibile

1. Scenario usa profili aggregati.
2. Report mostra messaggio.
3. Non genera tabella vuota ambigua.

---

#### A7 — Assumption snapshot mancante

1. Result non contiene snapshot.
2. Report mostra warning forte.
3. Scenario marcato needsReview.
4. Se possibile mostra pack corrente come riferimento non garantito.

---

#### A8 — Dati demo

1. Scenario è demo.
2. Report mostra badge Demo data.
3. Disclaimer rimane visibile.

---

#### A9 — Export/print con warning critici

1. Scenario ha warning critici.
2. L’utente prova a stampare.
3. Il sistema può mostrare conferma:
   “This report contains critical warnings.”
4. Se conferma, stampa con warning inclusi.

---

### 15. Validazioni

#### Validazioni bloccanti

* scenario mancante;
* company mancante;
* report view model non costruibile;
* dati numerici NaN;
* dati numerici Infinity;
* calculation result incoerente con scenario;
* assumption snapshot mancante in report reviewed, se configurato come bloccante;
* print non supportato dal browser, se azione richiesta;
* export report fallito.

---

#### Warning non bloccanti

* scenario non calcolato;
* scenario stale;
* scenario partial;
* scenario imported;
* scenario shared;
* scenario demo;
* funding mancante;
* costi N/A;
* budget non configurato;
* budget superato;
* pack custom;
* pack deprecated;
* assumption snapshot parziale;
* task breakdown non disponibile;
* warning critici;
* model restricted;

### 21. Criteri di accettazione

#### AC-001 — Apertura report

Given esiste uno scenario salvato,
when l’utente clicca “View report”,
then il sistema apre la pagina report scenario.

---

#### AC-002 — Executive summary

Given lo scenario ha calculation result valido,
when il report viene visualizzato,
then mostra monthly credits, monthly cost, annual cost, active users e budget status.

---

#### AC-003 — Range visibile

Given il risultato ha min/mid/max,
when il report mostra crediti o costi,
then il report include i tre valori o indica chiaramente il range.

---

#### AC-004 — Midpoint primario

Given il report mostra KPI,
when i valori sono range-based,
then il midpoint è il valore principale.

---

#### AC-005 — Scenario non calcolato

Given lo scenario non ha calculation result,
when l’utente apre il report,
then il report mostra input e messaggio “not calculated” senza valori finali.

---

#### AC-006 — Scenario stale

Given lo scenario ha result stale,
when il report viene aperto,
then mostra banner stale in alto.

---

#### AC-007 — Scenario invalid

Given lo scenario contiene errori bloccanti,
when il report viene aperto,
then non presenta numeri come finali e mostra errori.

---

#### AC-008 — Funding mancante

Given lo scenario ha crediti ma non funding,
when il report viene visualizzato,
then mostra crediti e costi N/A.

---

#### AC-009 — Budget non configurato

Given lo scenario non ha budget,
when il report viene visualizzato,
then mostra budget status Not configured.

---

#### AC-010 — Budget superato

Given lo scenario supera budget,
when il report viene visualizzato,
then mostra warning budget e overrun.

---

#### AC-011 — Segment breakdown

Given lo scenario ha segmenti calcolati,
when il report viene visualizzato,
then mostra tabella breakdown per segmento.

---

#### AC-012 — Intensity breakdown

Given lo scenario ha task light/medium/heavy,
when il report viene visualizzato,
then mostra breakdown per intensità.

---

#### AC-013 — Task breakdown disponibile

Given lo scenario usa task mix,
when il report viene visualizzato,
then mostra breakdown per task.

---

#### AC-014 — Task breakdown assente

Given lo scenario usa profili aggregati,
when il report viene visualizzato,
then mostra messaggio informativo e non una tabella vuota.

---

#### AC-015 — Model breakdown

Given lo scenario usa modelli,
when il report viene visualizzato,
then mostra breakdown per modello.

---

#### AC-016 — Assumption pack visibile

Given il report viene generato,
when l’utente visualizza la sezione assumptions,
then vede nome pack, versione e source date.

---

#### AC-017 — Custom assumptions

Given lo scenario usa valori custom,
when il report viene visualizzato,
then mostra badge o sezione custom assumptions.

---

#### AC-018 — Formula calcolo

Given il report viene visualizzato,
when l’utente apre la sezione “How this was calculated”,
then vede formula sintetica del calcolo.

---

#### AC-019 — Warning inclusi

Given lo scenario ha warning,
when il report viene visualizzato,
then i warning sono inclusi nella sezione dedicata.

---

#### AC-020 — Disclaimer visibile

Given il report mostra crediti o costi,
when il report viene visualizzato o stampato,
then include disclaimer sulle stime non ufficiali.

---

#### AC-021 — Print friendly

Given l’utente clicca Print / Save PDF,
when il browser apre la stampa,
then il layout è ottimizzato per PDF e non mostra controlli interattivi.

---

#### AC-022 — Export CSV da report

Given l’utente apre il report,
when clicca Export CSV,
then il sistema esporta i dati coerenti con il report.

---

#### AC-023 — Export JSON da report

Given l’utente apre il report,
when clicca Export JSON,
then il sistema esporta scenario e dipendenze.

---

#### AC-024 — Share link da report

Given lo scenario è condivisibile,
when l’utente clicca Share link,
then il sistema genera link o propone JSON se troppo grande.

---

#### AC-025 — Report da shared scenario

Given l’utente apre uno scenario condiviso,
when visualizza report,
then il report indica source shared e se non è salvato localmente.

---

#### AC-026 — Report da imported scenario

Given lo scenario è importato,
when visualizza report,
then il report indica source imported.

---

#### AC-027 — Demo data

Given lo scenario è demo,
when visualizza report,
then il report mostra badge demo data.

---

#### AC-028 — Coerenza con breakdown

Given il report mostra valori per segmento,
when confrontati con il breakdown risultati,
then i valori coincidono.

---

#### AC-029 — Coerenza con CSV

Given il report viene esportato anche in CSV,
when i valori sono confrontati,
then i KPI principali coincidono.

---

#### AC-030 — Nessuna modifica dati

Given l’utente apre o stampa il report,
when l’azione termina,
then scenario, calculation result e assumption snapshot non vengono modificati.

### 22. Priorità

Priorità funzionale: **Must have**

Il report scenario è il principale output finale dell’app.

Senza report, l’utente può calcolare e analizzare, ma non ha un artefatto chiaro da condividere, archiviare o discutere.

Il report è fondamentale per:

* presentazioni commerciali;
* validazione Finance;
* discussione con cliente;
* documentazione delle assunzioni;
* auditabilità;
* export;
* archiviazione;
* confronto tra scenari;
* governance.

### 23. Note UX

La UX del report deve essere chiara, professionale e leggibile.

Principi UX:

* executive summary in alto;
* KPI leggibili senza scroll eccessivo;
* range min/mid/max sempre chiaro;
* sezioni ordinate in modo logico;
* warning visibili ma non caotici;
* disclaimer sempre presente;
* tabelle compatte;
* azioni export disponibili;
* layout print-friendly;
* nessun controllo UI nella stampa;
* badge per stale, custom, imported, shared, demo;
* formula sintetica comprensibile;
* link al dettaglio “How this was calculated”;
* separazione netta tra crediti, costi, funding e budget.
