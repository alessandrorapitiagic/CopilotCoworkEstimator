## FP-013 — Salvataggio locale

---

### 1. Obiettivo funzionale

La funzionalità “Salvataggio locale” consente all’app di conservare nel browser dell’utente tutti i dati necessari al funzionamento del workbench di stima Microsoft Copilot Cowork Credits.

Poiché l’app è statica e non prevede backend, il salvataggio locale deve gestire:

* aziende;
* scenari;
* segmenti workforce;
* profili di utilizzo custom;
* task preset custom;
* modelli custom;
* assumption pack custom;
* funding plan;
* calculation result;
* comparison temporanee o salvate;
* preferenze utente;
* bozze wizard;
* dati importati;
* storico export;
* stato UI minimo.

L’obiettivo è permettere all’utente di chiudere e riaprire l’app ritrovando i propri dati, senza account e senza database remoto.

Il sistema deve anche offrire meccanismi di sicurezza funzionale:

* autosave;
* recovery bozza;
* validazione dati salvati;
* gestione schema version;
* migrazione dati;
* backup/export JSON;
* import JSON;
* rilevamento dati corrotti;
* reset controllato;
* avvisi su storage pieno o non disponibile.

Il salvataggio locale deve essere affidabile, ma deve sempre chiarire all’utente che i dati sono salvati nel browser e non su server.

---

### 2. Attori coinvolti

Gli attori principali sono:

* Consulente;
* Solution architect;
* Sales specialist;
* Account manager;
* Responsabile IT;
* Responsabile Finance;
* Cliente finale;
* Utente che lavora offline;
* Utente che usa browser aziendale;
* Utente che esporta backup;
* Utente che importa scenari da altri utenti.

---

### 3. Prerequisiti

Per usare il salvataggio locale devono essere disponibili:

* browser moderno;
* accesso a localStorage o IndexedDB;
* schema dati applicativo;
* storage service;
* validation service;
* import/export JSON;
* migration service;
* gestione errori;
* inizializzazione dati di sistema.

Per MVP è possibile usare `localStorage`.

Per versione avanzata è consigliabile valutare `IndexedDB` se il volume dati cresce molto.

---

### 4. Trigger

La funzionalità viene attivata quando:

* l’utente crea una company;
* l’utente modifica una company;
* l’utente crea uno scenario;
* l’utente modifica uno scenario;
* l’utente completa uno step del wizard;
* l’utente modifica segmenti;
* l’utente modifica usage profile;
* l’utente modifica task preset;
* l’utente modifica modelli;
* l’utente modifica assumption pack;
* l’utente modifica funding plan;
* il motore genera calculation result;
* l’utente salva una bozza;
* l’utente importa JSON;
* l’utente esporta JSON;
* l’utente apre l’app;
* l’utente ricarica la pagina;
* l’utente chiude o tenta di chiudere la pagina;
* il sistema rileva localStorage pieno;
* il sistema rileva dati corrotti;
* il sistema aggiorna lo schema dati.

---

### 5. Descrizione funzionale dettagliata

Il sistema deve salvare i dati dell’utente localmente nel browser.

Il salvataggio locale deve essere centralizzato in un servizio dedicato, non sparso nei componenti UI.

La UI non deve scrivere direttamente in localStorage. Deve chiamare servizi applicativi, che a loro volta usano lo storage service.

Il sistema deve prevedere:

1. **Store applicativo principale**

   * contiene tutti gli oggetti salvati dall’utente.

2. **Store dati di sistema**

   * contiene default iniziali, come profili standard, task preset standard, modelli standard e assumption pack di sistema.

3. **Store preferenze**

   * contiene preferenze UI, come currency default, ultimo scenario aperto, range mode preferito.

4. **Store bozze**

   * contiene wizard draft e autosave temporanei.

5. **Store metadata**

   * contiene schema version, last saved date, last export date, storage usage.

Il sistema deve caricare i dati all’avvio dell’app, validarli e, se necessario, migrarli alla versione corrente dello schema.

Se i dati non sono leggibili, l’app non deve bloccarsi. Deve mostrare un messaggio e proporre:

* tentare recovery;
* esportare dati grezzi;
* resettare dati locali;
* importare backup.

### 9. Regole di business

#### BR-FP013-001 — Salvataggio locale obbligatorio

Tutti i dati utente devono essere salvati localmente.

Nessun dato deve richiedere backend per essere persistito.

---

#### BR-FP013-002 — Separazione dati system e custom

I dati di sistema devono essere distinguibili dai dati custom.

Esempi:

* usage profile standard;
* task preset standard;
* modelli standard;
* assumption pack system.

I dati system possono essere rigenerati in caso di perdita.

I dati custom devono essere preservati.

---

#### BR-FP013-003 — Dati system non eliminabili

Il sistema non deve consentire l’eliminazione definitiva dei dati system.

Se mancano, devono essere reinizializzati.

---

#### BR-FP013-004 — Autosave wizard

Durante il wizard, il sistema deve salvare automaticamente una bozza.

L’autosave deve avvenire:

* al cambio step;
* dopo modifica significativa;
* prima di uscire dal wizard;
* a intervallo controllato, se previsto.

---

#### BR-FP013-005 — Salvataggio esplicito scenario

Uno scenario deve essere salvato esplicitamente quando l’utente clicca:

* Save draft;
* Save as reviewed;
* Save scenario;
* Duplicate scenario;
* Import scenario.

---

#### BR-FP013-006 — Calculation result salvato con scenario

Quando uno scenario viene salvato come reviewed, il calculation result valido deve essere salvato insieme allo scenario o referenziato in modo stabile.

---

#### BR-FP013-007 — Snapshot delle assunzioni

Ogni calculation result salvato deve includere snapshot o riferimento stabile alle assunzioni usate.

Questo evita che risultati storici cambino dopo aggiornamenti dei pack.

---

#### BR-FP013-008 — Schema version obbligatoria

Lo stato salvato deve sempre avere una `schemaVersion`.

Se manca, il sistema deve trattare i dati come legacy.

---

#### BR-FP013-009 — Migrazione dati

Se la versione schema salvata è precedente alla versione corrente, il sistema deve eseguire una migrazione.

La migrazione deve:

* non cancellare dati utente;
* aggiungere default mancanti;
* normalizzare campi;
* generare warning se qualcosa non è migrabile.

---

#### BR-FP013-010 — Nessuna migrazione distruttiva automatica

Il sistema non deve eliminare dati utente durante migrazione automatica, salvo dati chiaramente corrotti e non recuperabili.

---

#### BR-FP013-011 — Backup prima di migrazione

Prima di una migrazione significativa, il sistema deve poter creare un backup interno o proporre export.

Per MVP può bastare conservare una copia temporanea nello storage.

---

#### BR-FP013-012 — Dati corrotti

Se il JSON salvato non è parsabile o non rispetta lo schema minimo, il sistema deve:

* non crashare;
* mostrare errore;
* proporre recovery;
* proporre reset;
* proporre import backup.

---

#### BR-FP013-013 — Recovery dati parziale

Se solo alcune entità sono corrotte, il sistema deve tentare recovery parziale.

Esempio:

* company valide recuperate;
* scenario corrotti scartati o marcati invalid;
* profili system rigenerati.

---

#### BR-FP013-014 — localStorage non disponibile

Se localStorage non è disponibile, il sistema deve usare fallback memory.

Il fallback memory:

* funziona solo nella sessione corrente;
* non persiste dopo refresh;
* deve mostrare warning.

---

#### BR-FP013-015 — localStorage pieno

Se localStorage è pieno, il sistema deve:

* bloccare salvataggio;
* mostrare warning;
* proporre export JSON;
* proporre cancellazione dati non necessari;
* proporre archiviazione o reset.

---

#### BR-FP013-016 — Export backup

L’utente deve poter esportare tutti i dati locali in JSON.

Il file esportato deve contenere:

* schemaVersion;
* dati utente;
* dati custom;
* scenari;
* calculation results;
* assumption pack custom;
* metadata.

---

#### BR-FP013-017 — Import backup

L’utente deve poter importare un file JSON precedentemente esportato.

L’import deve validare dati e schema.

---

#### BR-FP013-018 — Merge o replace import

L’import deve supportare almeno una modalità.

Per MVP si consiglia:

* import as new;
* oppure replace all solo con conferma forte.

Modalità avanzate:

* merge;
* replace selected;
* import only scenarios;
* import only assumptions.

---

#### BR-FP013-019 — Conferma reset

Il reset dei dati locali deve richiedere conferma forte.

Esempio:

“Scrivi RESET per confermare.”

---

#### BR-FP013-020 — Privacy locale

La UI deve chiarire che i dati sono salvati localmente nel browser.

Nessun dato viene inviato a server dall’app statica.

---

#### BR-FP013-021 — Device/browser specific

Il sistema deve spiegare che i dati locali sono legati a:

* browser;
* dispositivo;
* profilo utente;
* eventuale cancellazione cache.

---

#### BR-FP013-022 — Export consigliato

Se l’utente ha dati importanti o molti scenari, il sistema deve suggerire export periodico.

---

#### BR-FP013-023 — Shared link non salva automaticamente

Uno scenario aperto da link condiviso non deve essere salvato automaticamente nella libreria locale, salvo azione esplicita dell’utente.

---

#### BR-FP013-024 — Import non sovrascrive senza conferma

L’import non deve sovrascrivere dati locali senza conferma esplicita.

---

#### BR-FP013-025 — Validazione prima del salvataggio

Prima di salvare, il sistema deve validare struttura minima dei dati.

---

#### BR-FP013-026 — Debounce autosave

L’autosave deve essere debounced per evitare scritture eccessive.

Esempio:

* 500 ms;
* 1 secondo;
* oppure al cambio step.

---

#### BR-FP013-027 — Atomic save

Il salvataggio dello stato principale deve essere il più possibile atomico.

Per localStorage, il sistema può:

* serializzare intero stato;
* scrivere in chiave temporanea;
* validare;
* sostituire chiave principale.

---

#### BR-FP013-028 — Last saved indicator

La UI deve mostrare quando i dati sono stati salvati.

Esempi:

* “Saved locally”;
* “Last saved 14:32”;
* “Unsaved changes”;
* “Autosave failed”.

---

#### BR-FP013-029 — Dati demo

I dati demo devono essere distinguibili dai dati reali.

L’utente deve poter rimuovere o duplicare dati demo.

---

#### BR-FP013-030 — Nessun dato sensibile richiesto

L’app non deve richiedere dati personali sensibili per funzionare.

Campi come company name e notes sono inseriti volontariamente dall’utente.

---

### 10. Logiche di salvataggio

## 10.1 Salvataggio stato principale

```text id="8qj8qa"
saveAppState(appState):
  validateAppState(appState)
  serialized = JSON.stringify(appState)
  localStorage.setItem("coworkEstimatorAppState.tmp", serialized)
  verify tmp write
  localStorage.setItem("coworkEstimatorAppState", serialized)
  localStorage.removeItem("coworkEstimatorAppState.tmp")
```

---

## 10.2 Caricamento stato

```text id="rw4v8v"
loadAppState():
  raw = localStorage.getItem("coworkEstimatorAppState")
  if raw is null:
    return initializeDefaultState()

  parsed = JSON.parse(raw)
  migrated = migrateIfNeeded(parsed)
  validated = validateLoadedState(migrated)

  return validated
```

---

## 10.3 Fallback su chiave temporanea

```text id="zvbzlc"
if main state is corrupted
  and tmp state exists
  and tmp state is valid:
    recover tmp state
```

---

## 10.4 Autosave wizard

```text id="w5dy4r"
onWizardStateChange:
  debounce(1000ms):
    saveWizardDraft(wizardState)
```

---

## 10.5 Rilevamento stale result

```text id="a1kf3f"
if scenario.updatedAt > calculationResult.calculatedAt:
  calculationResult.status = "stale"
```

Oppure con hash:

```text id="g6i4fu"
if calculationResult.inputHash != calculateInputHash(currentInputs):
  calculationResult.status = "stale"
```

---

## 10.6 Storage usage

```text id="4dlsje"
storageUsageBytes =
  new Blob([JSON.stringify(appState)]).size
```

Il sistema può mostrare warning se supera una soglia.

### 11. Flusso principale — Primo accesso

1. L’utente apre l’app.
2. Il sistema cerca dati locali.
3. Non trova dati.
4. Inizializza stato default.
5. Crea dati system:

   * profili standard;
   * task preset standard;
   * modelli standard;
   * assumption pack default.
6. Mostra dashboard vuota.
7. Propone:

   * creare company;
   * importare backup;
   * caricare dati demo.

---

### 12. Flusso principale — Salvataggio scenario

1. L’utente crea o modifica scenario.
2. Il sistema aggiorna stato applicativo in memoria.
3. L’utente clicca Save o Save as reviewed.
4. Il sistema valida scenario.
5. Il sistema esegue calculation result, se necessario.
6. Il sistema salva:

   * scenario;
   * segmenti;
   * funding plan;
   * calculation result;
   * assumption snapshot;
   * metadata.
7. Il sistema aggiorna `updatedAt`.
8. Il sistema scrive in localStorage.
9. La UI mostra “Saved locally”.

---

### 13. Flusso principale — Autosave bozza wizard

1. L’utente entra nel wizard.
2. Il sistema crea draft ID.
3. L’utente compila gli step.
4. A ogni modifica rilevante, il sistema salva bozza.
5. Se la pagina viene ricaricata, il sistema rileva bozza.
6. Mostra dialog:

   * Resume draft;
   * Discard draft;
   * Export draft.
7. L’utente sceglie.
8. Il wizard viene ripristinato o scartato.

---

### 14. Flusso principale — Export backup

1. L’utente apre Settings o Data management.
2. Clicca “Export backup”.
3. Il sistema valida stato corrente.
4. Crea payload JSON.
5. Include metadata:

   * schemaVersion;
   * exportedAt;
   * appVersion;
   * counts;
   * custom assumptions.
6. Il file viene scaricato.
7. `lastExportAt` viene aggiornato.
8. La UI mostra conferma.

---

### 15. Flusso principale — Import backup

1. L’utente clicca “Import backup”.
2. Seleziona file JSON.
3. Il sistema legge file.
4. Valida JSON.
5. Mostra preview:

   * aziende;
   * scenari;
   * custom packs;
   * profili custom;
   * eventuali warning.
6. L’utente sceglie modalità import.
7. Il sistema importa dati.
8. Esegue migrazione se necessario.
9. Salva stato locale.
10. Mostra risultato import.

---

### 16. Flussi alternativi

#### A1 — localStorage non disponibile

1. L’app prova a leggere localStorage.
2. Il browser blocca accesso.
3. Il sistema usa memory storage.
4. Mostra banner:
   “Local storage is not available. Data will be lost after refresh.”
5. L’utente può continuare.
6. Export JSON viene raccomandato.

---

#### A2 — localStorage pieno

1. L’utente prova a salvare.
2. localStorage genera quota exceeded.
3. Il sistema intercetta errore.
4. Mostra messaggio.
5. Propone:

   * export backup;
   * eliminare dati demo;
   * eliminare scenari archiviati;
   * reset locale.

---

#### A3 — Dati corrotti all’avvio

1. L’app legge localStorage.
2. JSON.parse fallisce.
3. Il sistema non crasha.
4. Mostra recovery screen.
5. L’utente può:

   * scaricare dati grezzi;
   * provare recovery;
   * resettare;
   * importare backup.

---

#### A4 — Migrazione schema

1. L’app carica dati con schema vecchio.
2. Migration service identifica versione.
3. Esegue migrazioni incrementali.
4. Valida risultato.
5. Salva stato aggiornato.
6. Mostra warning se necessario.

---

#### A5 — Import con dati duplicati

1. L’utente importa backup con company/scenari già presenti.
2. Il sistema rileva conflitti ID.
3. Per MVP genera nuovi ID.
4. Mostra mapping import.
5. Salva dati come nuovi.

---

#### A6 — Replace all import

1. L’utente sceglie modalità Replace all.
2. Il sistema mostra conferma forte.
3. L’utente conferma.
4. Il sistema sostituisce dati locali.
5. Dati system vengono reinizializzati se mancanti.

---

#### A7 — Shared link

1. L’utente apre scenario da link.
2. Il sistema decodifica payload.
3. Mostra preview.
4. Non salva automaticamente.
5. L’utente clicca “Save locally”.
6. Lo scenario viene salvato con nuovo ID.

---

#### A8 — Reset dati locali

1. L’utente apre Data management.
2. Clicca Reset local data.
3. Il sistema chiede conferma forte.
4. L’utente conferma.
5. Il sistema cancella dati custom.
6. Reinizializza dati system.
7. Mostra dashboard vuota.

---

### 17. Validazioni

#### Validazioni bloccanti

* appState non parsabile;
* schemaVersion mancante e migrazione impossibile;
* struttura root non valida;
* companies non array;
* scenarios non array;
* assumptionPacks non array;
* scenario senza ID;
* company senza ID;
* ID duplicati non risolti;
* import file non JSON;
* import schema non supportato;
* storage write fallito;
* dati system minimi non inizializzabili.

---

#### Warning non bloccanti

* localStorage non disponibile;
* salvataggio in memory storage;
* storage quasi pieno;
* dati legacy migrati;
* alcuni oggetti importati richiedono review;
* scenario importato con fallback Auto;
* assumption pack custom importato;
* dati demo presenti;
* ultimo export assente o vecchio;
* calculation result stale dopo import;
* scenario draft non completato;
* shared scenario non salvato;
* data health warning;
* recovery parziale eseguita.

### 22. Edge case

### 23. Criteri di accettazione

#### AC-001 — Primo accesso

Given l’utente apre l’app senza dati locali,
when l’app viene inizializzata,
then il sistema crea stato default e mostra dashboard vuota.

---

#### AC-002 — Dati system inizializzati

Given l’app viene inizializzata,
when non esistono dati system,
then il sistema crea profili, task, modelli e assumption pack standard.

---

#### AC-003 — Salvataggio company

Given l’utente crea una company,
when salva,
then la company viene persistita localmente.

---

#### AC-004 — Recupero dopo refresh

Given l’utente ha salvato una company,
when ricarica la pagina,
then la company è ancora disponibile.

---

#### AC-005 — Salvataggio scenario

Given l’utente salva uno scenario,
when il salvataggio termina,
then scenario, segmenti, funding plan e calculation result vengono persistiti.

---

#### AC-006 — Autosave wizard

Given l’utente compila il wizard,
when modifica dati rilevanti,
then il sistema salva automaticamente una bozza locale.

---

#### AC-007 — Recovery bozza

Given esiste una wizard draft autosalvata,
when l’utente riapre l’app,
then il sistema propone di riprenderla o scartarla.

---

#### AC-008 — localStorage non disponibile

Given localStorage non è disponibile,
when l’app si avvia,
then il sistema usa fallback memory e mostra warning.

---

#### AC-009 — localStorage pieno

Given localStorage è pieno,
when il sistema prova a salvare,
then mostra errore e propone export o pulizia dati.

---

#### AC-010 — Export backup

Given esistono dati locali,
when l’utente clicca Export backup,
then il sistema genera un file JSON con tutti i dati necessari.

---

#### AC-011 — Import backup valido

Given l’utente seleziona un backup JSON valido,
when conferma import,
then i dati vengono importati e salvati localmente.

---

#### AC-012 — Import backup non valido

Given l’utente seleziona un file non valido,
when il sistema prova a importarlo,
then mostra errore e non modifica i dati locali.

---

#### AC-013 — Import senza sovrascrittura automatica

Given esistono dati locali,
when l’utente importa un backup,
then il sistema non sovrascrive dati esistenti senza conferma.

---

#### AC-014 — Dati corrotti

Given i dati locali sono corrotti,
when l’app si avvia,
then il sistema non crasha e mostra recovery screen.

---

#### AC-015 — Recovery parziale

Given alcuni dati sono corrotti e altri validi,
when il sistema esegue recovery,
then recupera gli oggetti validi e segnala quelli non recuperati.

---

#### AC-016 — Migrazione schema

Given i dati locali hanno schemaVersion precedente,
when l’app si avvia,
then il sistema migra i dati alla versione corrente.

---

#### AC-017 — Migrazione non distruttiva

Given una migrazione viene eseguita,
when termina,
then i dati utente validi non vengono cancellati.

---

#### AC-018 — Reset locale

Given l’utente apre Data management,
when conferma reset con conferma forte,
then il sistema cancella i dati custom e reinizializza dati system.

---

#### AC-019 — Reset non confermato

Given l’utente apre reset locale,
when non conferma correttamente,
then il sistema non cancella dati.

---

#### AC-020 — Last saved indicator

Given l’utente salva dati,
when il salvataggio va a buon fine,
then la UI mostra indicazione di salvataggio locale.

---

#### AC-021 — Shared link non autosalvato

Given l’utente apre uno scenario da link condiviso,
when visualizza la preview,
then lo scenario non viene salvato localmente finché l’utente non clicca Save locally.

---

#### AC-022 — Calculation result stale

Given uno scenario salvato ha input modificati,
when l’app carica lo scenario,
then il calculation result viene marcato come stale.

---

#### AC-023 — Dati demo distinguibili

Given l’utente carica dati demo,
when la dashboard li mostra,
then sono marcati come demo e distinguibili dai dati reali.

---

#### AC-024 — Export history

Given l’utente esporta un backup,
when l’export è completato,
then `lastExportAt` viene aggiornato.

---

#### AC-025 — Privacy locale

Given l’utente apre Data management,
when visualizza le informazioni storage,
then il sistema chiarisce che i dati sono salvati nel browser e non sincronizzati.

### 24. Priorità

Priorità funzionale: **Must have**

Il salvataggio locale è indispensabile perché l’app è statica e senza backend.

Senza questa funzionalità:

* ogni refresh perderebbe i dati;
* non sarebbe possibile gestire portfolio aziende;
* non sarebbe possibile salvare scenari;
* non sarebbe possibile confrontare scenari nel tempo;
* non sarebbe possibile import/export affidabile;
* non sarebbe possibile creare un workbench consulenziale.

---

### 25. Note UX

La UX del salvataggio locale deve generare fiducia.

Principi UX:

* mostrare chiaramente quando i dati sono salvati;
* spiegare che il salvataggio è locale;
* non spaventare l’utente con dettagli tecnici;
* suggerire export periodico;
* offrire recovery invece di errore generico;
* non cancellare mai dati senza conferma forte;
* rendere import/export facile;
* mostrare preview prima di import;
* distinguere dati reali, demo, importati e condivisi;
* mostrare warning storage solo quando utile;
* evitare autosave troppo invasivo;
* proteggere l’utente da perdita dati durante wizard.

### 27. Definizione di completato

Il FP-013 può essere considerato completo quando:

* l’app inizializza dati default al primo accesso;
* company e scenari vengono salvati localmente;
* segmenti e funding plan vengono salvati localmente;
* calculation result viene salvato;
* usage profile custom vengono salvati;
* task preset custom vengono salvati;
* modelli custom vengono salvati;
* assumption pack custom vengono salvati;
* i dati persistono dopo refresh;
* il wizard usa autosave;
* le bozze sono recuperabili;
* lo storage ha schema version;
* i dati legacy vengono migrati;
* localStorage non disponibile è gestito;
* localStorage pieno è gestito;
* dati corrotti non causano crash;
* recovery parziale è supportata;
* export backup JSON è disponibile;
* import backup JSON è disponibile;
* import non sovrascrive senza conferma;
* reset locale richiede conferma forte;
* dati system possono essere reinizializzati;
* shared link non salva automaticamente;
* la UI mostra stato salvataggio;
* la UI chiarisce che i dati sono locali;
* tutto funziona senza backend e resta compatibile con GitHub Pages.
