## FP-014 — Export CSV

---

### 1. Obiettivo funzionale

La funzionalità “Export CSV” consente all’utente di esportare in formato CSV i dati principali generati dall’app di stima Microsoft Copilot Cowork Credits.

L’obiettivo è permettere all’utente di usare i dati fuori dall’app, ad esempio in:

* Microsoft Excel;
* Power BI;
* Google Sheets;
* sistemi CRM;
* presentazioni commerciali;
* business case;
* analisi Finance;
* confronti interni;
* documenti di prevendita;
* reportistica cliente.

L’export CSV deve essere semplice, leggibile e coerente con i risultati mostrati in app.

La funzionalità deve supportare più livelli di export:

1. **Scenario summary CSV**

   * una riga per scenario.

2. **Scenario line-item CSV**

   * una riga per segmento, task o breakdown item.

3. **Segment breakdown CSV**

   * dettaglio per workforce segment.

4. **Task breakdown CSV**

   * dettaglio per task preset, se disponibile.

5. **Model breakdown CSV**

   * dettaglio per modello AI.

6. **Funding breakdown CSV**

   * dettaglio di copertura economica.

7. **Comparison CSV**

   * confronto tra due o più scenari.

8. **Portfolio CSV**

   * aggregazione multi-company e multi-scenario.

Il CSV deve esportare dati di stima, non deve presentare i valori come prezzi ufficiali o contrattuali.

---

### 2. Attori coinvolti

Gli attori principali sono:

* Consulente;
* Solution architect;
* Sales specialist;
* Account manager;
* Responsabile Finance;
* Responsabile Procurement;
* Responsabile IT;
* Responsabile AI adoption;
* Cliente finale;
* Utente che prepara business case;
* Utente che deve condividere dati con team esterni;
* Utente che importa i risultati in Excel o Power BI.

---

### 3. Prerequisiti

Per eseguire un export CSV devono essere disponibili:

* almeno uno scenario salvato;
* calculation result valido o parziale;
* company associata;
* assumption pack associato;
* funding plan, se si esportano costi;
* breakdown disponibili;
* servizio export;
* servizio formatting;
* dati coerenti nello storage locale.

Per alcuni export specifici servono prerequisiti aggiuntivi.

| Tipo export       | Prerequisito                       |
| ----------------- | ---------------------------------- |
| Scenario summary  | Scenario calcolato o salvato       |
| Segment breakdown | Breakdown per segmento disponibile |
| Task breakdown    | Task mix o task preset disponibili |
| Model breakdown   | Breakdown per modello disponibile  |
| Funding breakdown | Funding result disponibile         |
| Comparison        | Almeno due scenari confrontabili   |
| Portfolio         | Almeno una company con scenario    |

---

### 4. Trigger

La funzionalità viene attivata quando:

* l’utente clicca “Export CSV” da scenario report;
* l’utente clicca “Export” dalla review del wizard;
* l’utente esporta il breakdown risultati;
* l’utente esporta una comparison;
* l’utente esporta il portfolio;
* l’utente esporta una company con tutti gli scenari;
* l’utente esporta risultati filtrati;
* l’utente vuole analizzare dati in Excel;
* l’utente vuole condividere dati con Finance;
* l’utente vuole creare un backup tabellare non completo.

---

### 5. Descrizione funzionale dettagliata

Il sistema deve generare file CSV scaricabili direttamente dal browser, senza backend.

L’utente deve poter scegliere quale export generare.

Tipi minimi per MVP:

1. Scenario line-item CSV.
2. Portfolio CSV.
3. Comparison CSV.

Tipi consigliati per versione completa:

1. Scenario summary CSV.
2. Segment breakdown CSV.
3. Task breakdown CSV.
4. Model breakdown CSV.
5. Funding breakdown CSV.
6. Assumption summary CSV.
7. Warning CSV.

L’export deve includere valori numerici già calcolati dal motore, non ricalcolarli in modo indipendente.

Il CSV deve essere generato a partire da:

* scenario;
* calculation result;
* breakdown;
* funding result;
* assumption snapshot;
* warning;
* metadata.

Ogni file CSV deve contenere:

* intestazioni leggibili;
* valori coerenti con la UI;
* valori min/mid/max, dove applicabile;
* date in formato standard;
* valuta;
* assumption pack name/version;
* disclaimer o nota di stima, se tecnicamente gestibile;
* identificativi stabili per scenario/company.

### 7. Regole di business

#### BR-FP014-001 — Export browser-only

L’export CSV deve essere generato interamente nel browser.

Non deve richiedere backend, API server o database remoto.

---

#### BR-FP014-002 — Nessun ricalcolo indipendente

L’export CSV deve usare i dati già prodotti dal motore di calcolo.

Non deve implementare formule parallele che possano divergere dalla UI.

---

#### BR-FP014-003 — Export coerente con range

Gli export devono includere min, mid e max dove disponibili.

Se l’utente esporta una vista filtrata su un range specifico, il file deve indicare il range mode.

---

#### BR-FP014-004 — Midpoint sempre presente

Il valore midpoint deve essere sempre presente negli export principali, se disponibile.

---

#### BR-FP014-005 — Costi opzionali se funding mancante

Se uno scenario ha crediti ma non funding valido, il CSV deve esportare i crediti e lasciare i costi vuoti o `N/A`.

---

#### BR-FP014-006 — Scenario stale

Se il risultato è stale, il CSV deve includere una colonna che lo indichi.

Esempio:

```text id="gtx68x"
calculation_status = stale
```

---

#### BR-FP014-007 — Scenario invalid

Uno scenario invalid può essere esportato solo come dati input o con valori risultato vuoti.

Non deve esportare numeri incompleti come validi.

---

#### BR-FP014-008 — Assumption metadata obbligatori

Ogni export scenario deve includere:

* assumption pack name;
* assumption pack version;
* source date;
* custom/deprecated flag.

---

#### BR-FP014-009 — Funding metadata obbligatori

Se sono esportati costi, il CSV deve includere:

* funding mode;
* currency;
* price per credit;
* effective price per credit, se applicabile.

---

#### BR-FP014-010 — Disclaimer o export note

Il CSV deve includere, dove possibile, una riga o colonna di nota che chiarisca che i valori sono stime basate su assunzioni.

Per semplicità, l’app può includere una colonna:

```text id="s2qv9l"
estimation_disclaimer
```

---

#### BR-FP014-011 — Nomi file leggibili

Il nome file deve essere descrittivo.

Esempi:

```text id="n3bsz4"
cowork-scenario-acme-full-rollout-2026-06-24.csv
cowork-portfolio-export-2026-06-24.csv
cowork-comparison-acme-2026-06-24.csv
```

---

#### BR-FP014-012 — Sanitizzazione nome file

Il sistema deve rimuovere o sostituire caratteri non validi nei nomi file.

---

#### BR-FP014-013 — Encoding UTF-8

Il CSV deve essere generato in UTF-8.

Per compatibilità Excel, può includere BOM UTF-8.

---

#### BR-FP014-014 — Delimitatore configurabile

Per MVP il delimitatore può essere virgola.

Opzione avanzata:

* comma;
* semicolon.

In contesti europei, il semicolon può essere utile.

---

#### BR-FP014-015 — Numeri esportati come valori numerici

I valori numerici devono essere esportati come numeri, non come stringhe formattate con simboli valuta.

Esempio corretto:

```text id="5kxdyi"
monthly_cost_mid = 1234.56
currency = EUR
```

Esempio da evitare:

```text id="kgrc3g"
monthly_cost_mid = €1,234.56
```

---

#### BR-FP014-016 — Date in formato ISO

Le date devono essere esportate in formato ISO.

Esempio:

```text id="n6xqfb"
2026-06-24T10:30:00Z
```

---

#### BR-FP014-017 — Valori null

I valori non disponibili devono essere esportati come:

* vuoto;
* oppure `N/A`.

Per MVP si consiglia campo vuoto, più facile da gestire in Excel.

---

#### BR-FP014-018 — Campi multilinea

Campi note o disclaimer con newline devono essere sanificati o correttamente quoted.

---

#### BR-FP014-019 — Separazione export CSV e JSON

Il CSV serve per analisi tabellare.

Non deve sostituire JSON export/import, che resta il formato per backup completo e portabilità scenario.

---

#### BR-FP014-020 — Export non modifica dati

L’azione di export non deve modificare scenario, company o risultati.

Può aggiornare solo metadata come `lastExportAt`.

---

#### BR-FP014-021 — Export history

Dopo export riuscito, il sistema può salvare:

* tipo export;
* data export;
* scenario o portfolio esportato;
* nome file.

---

#### BR-FP014-022 — Filtri applicati

Se l’utente esporta dati filtrati, il CSV deve includere metadata o filename che indichino il filtro applicato.

---

#### BR-FP014-023 — Export dati demo

Se l’export include dati demo, il CSV deve indicarlo con una colonna `is_demo`.

---

#### BR-FP014-024 — Export custom assumptions

Se il risultato usa custom assumptions, il CSV deve includere un flag e possibilmente i principali valori custom.

---

#### BR-FP014-025 — Export warning

Gli export principali devono includere almeno:

* warning count;
* critical warning count;
* warning summary.

Un export warning dedicato può includere il dettaglio completo.

### 8. Logiche di generazione CSV

## 8.1 Pipeline export

```text id="3qf6vq"
exportCsv(exportType, sourceData, options):
  validateExportRequest()
  buildRows()
  normalizeValues()
  escapeCsvValues()
  buildCsvString()
  createBlob()
  triggerDownload()
  updateExportHistory()
```

---

## 8.2 Escaping CSV

I valori devono essere quotati se contengono:

* virgola;
* punto e virgola;
* newline;
* virgolette;
* caratteri speciali.

```text id="2uahbk"
escapeCsvValue(value):
  if value contains delimiter or quote or newline:
    return quote(value.replace('"', '""'))
  else:
    return value
```

---

## 8.3 Generazione filename

```text id="gdqrmy"
filename =
  "cowork-" +
  exportType +
  "-" +
  sanitize(companyName or "portfolio") +
  "-" +
  currentDate +
  ".csv"
```

### 9. Flusso principale — Export scenario line-item CSV

1. L’utente apre uno scenario calcolato.
2. Clicca “Export CSV”.
3. Il sistema mostra opzioni export.
4. L’utente seleziona “Scenario line-item”.
5. Il sistema recupera calculation result.
6. Il sistema recupera breakdown per segmento.
7. Il sistema costruisce righe CSV.
8. Il sistema include metadata scenario e assumption.
9. Il sistema genera file.
10. Il browser avvia download.
11. Il sistema aggiorna export history.
12. La UI mostra conferma.

---

### 10. Flusso principale — Export portfolio CSV

1. L’utente apre portfolio view.
2. Applica eventuali filtri.
3. Clicca “Export portfolio CSV”.
4. Il sistema recupera company e scenari inclusi.
5. Aggrega valori per company e scenario.
6. Costruisce righe CSV.
7. Genera file.
8. Avvia download.
9. Mostra conferma.

---

### 11. Flusso principale — Export comparison CSV

1. L’utente apre scenario comparison.
2. Clicca “Export comparison CSV”.
3. Il sistema recupera comparison result.
4. Costruisce righe metriche e delta.
5. Include baseline scenario e range mode.
6. Include warning su pack/funding diversi.
7. Genera CSV.
8. Avvia download.

### 12. Flussi alternativi

#### A1 — Scenario senza calculation result

1. L’utente prova a esportare scenario.
2. Il sistema rileva assenza calculation result.
3. Mostra messaggio.
4. Propone:

   * calcolare scenario;
   * esportare solo input;
   * annullare.

---

#### A2 — Scenario stale

1. L’utente esporta scenario con risultato stale.
2. Il sistema mostra warning.
3. L’utente può:

   * ricalcolare prima di esportare;
   * esportare risultato salvato marcato stale;
   * annullare.
4. Se esporta, il CSV include `calculation_status = stale`.

---

#### A3 — Funding mancante

1. Lo scenario ha crediti ma non costi.
2. Il sistema esporta crediti.
3. Le colonne costo restano vuote.
4. Il CSV include warning funding mancante.

---

#### A4 — Task breakdown non disponibile

1. L’utente seleziona export task breakdown.
2. Lo scenario non usa task mix.
3. Il sistema mostra messaggio.
4. Propone export segment breakdown.

---

#### A5 — Dataset vuoto

1. L’utente prova a esportare portfolio senza scenari.
2. Il sistema blocca export.
3. Mostra messaggio:
   “No scenarios available to export.”

---

#### A6 — Errore generazione file

1. Il sistema prova a creare CSV.
2. Si verifica errore.
3. Il sistema mostra messaggio.
4. Nessun dato viene modificato.

---

#### A7 — CSV molto grande

1. L’utente esporta portfolio molto ampio.
2. Il sistema genera file.
3. Se il browser rallenta, mostra stato loading.
4. Se fallisce, suggerisce export filtrato.

---

#### A8 — Delimitatore semicolon

1. L’utente sceglie formato europeo.
2. Il sistema usa `;` come delimitatore.
3. I decimali restano con punto o seguono opzione locale, se implementata.
4. Il file viene generato.

---

#### A9 — Export con dati demo

1. L’utente esporta portfolio con dati demo.
2. Il sistema include colonna `is_demo`.
3. Eventuale warning mostra che il file contiene dati demo.

---

#### A10 — Export comparison con scenario invalid

1. La comparison contiene scenario invalid.
2. Il sistema esporta solo scenari comparabili.
3. Include riga o nota per scenario escluso.
4. Mostra warning.

### 13. Validazioni

#### Validazioni bloccanti

* tipo export mancante;
* tipo export non supportato;
* scenario inesistente;

### 19. Criteri di accettazione

#### AC-001 — Export scenario line-item

Given uno scenario calcolato valido,
when l’utente seleziona Scenario line-item CSV,
then il sistema genera un CSV con una riga per segmento o item di calcolo.

---

#### AC-002 — Colonne scenario line-item

Given viene esportato scenario line-item,
when il file viene generato,
then include company, scenario, segment, headcount, adoption, task, model, credits e costs.

---

#### AC-003 — Export portfolio

Given esistono company con scenari,
when l’utente esporta portfolio CSV,
then il sistema genera un CSV aggregato per company e scenario.

---

#### AC-004 — Export comparison

Given esiste una comparison valida,
when l’utente esporta comparison CSV,
then il sistema include baseline, scenario confrontati, metriche, delta assoluti e delta percentuali.

---

#### AC-005 — Min/mid/max inclusi

Given lo scenario ha risultati range-based,
when viene esportato CSV,
then il file include valori min, mid e max.

---

#### AC-006 — Funding mancante

Given uno scenario ha crediti ma non funding valido,
when viene esportato CSV,
then le colonne costo sono vuote e il file include warning funding mancante.

---

#### AC-007 — Scenario stale

Given uno scenario ha calculation result stale,
when l’utente esporta comunque,
then il CSV include una colonna che indica stato stale.

---

#### AC-008 — Scenario invalid

Given uno scenario è invalid,
when l’utente prova a esportare risultati,
then il sistema blocca export risultato o esporta solo input chiaramente marcati.

---

#### AC-009 — Task breakdown non disponibile

Given lo scenario non usa task mix,
when l’utente prova a esportare task breakdown,
then il sistema mostra messaggio e non genera file vuoto ambiguo.

---

#### AC-010 — Assumption metadata

Given viene esportato uno scenario,
when il CSV viene generato,
then include assumption pack name, version e source date.

---

#### AC-011 — Funding metadata

Given vengono esportati costi,
when il CSV viene generato,
then include funding mode, currency e price per credit.

---

#### AC-012 — CSV UTF-8

Given viene generato un CSV,
when il file viene aperto,
then supporta caratteri UTF-8.

---

#### AC-013 — Escaping valori

Given un campo contiene virgole, virgolette o newline,
when viene esportato CSV,
then il valore viene correttamente quoted.

---

#### AC-014 — Filename leggibile

Given l’utente esporta uno scenario,
when il file viene generato,
then il nome file contiene tipo export, nome scenario/company e data.

---

#### AC-015 — Export non modifica scenario

Given l’utente esporta CSV,
when il download termina,
then scenario e calculation result non vengono modificati.

---

#### AC-016 — Export history

Given l’export va a buon fine,
when il sistema aggiorna metadata,
then `lastExportAt` viene aggiornato.

---

#### AC-017 — Dati demo

Given l’export include dati demo,
when il CSV viene generato,
then include colonna `is_demo`.

---

#### AC-018 — Portfolio vuoto

Given non esistono scenari nel portfolio,
when l’utente prova a esportare,
then il sistema mostra errore e non genera CSV vuoto.

---

#### AC-019 — Delimitatore configurabile

Given l’utente seleziona semicolon come delimitatore,
when il CSV viene generato,
then le colonne sono separate da `;`.

---

#### AC-020 — Disclaimer

Given l’utente esporta valori economici,
when il CSV viene generato,
then il file include una nota o colonna che indica che i valori sono stime basate su assunzioni.

### 20. Priorità

Priorità funzionale: **Must have**

L’export CSV è indispensabile perché consente di portare i risultati fuori dall’app.

Senza CSV, l’app resta utile solo come interfaccia interattiva.

Con CSV, invece, diventa utilizzabile in processi reali:

* analisi Excel;
* presentazioni commerciali;
* business case Finance;
* pipeline portfolio;
* confronto scenari;
* lavoro consulenziale;
* allegati a email o documenti.

---

### 21. Note UX

La UX dell’export CSV deve essere semplice.

Principi UX:

* un pulsante export sempre visibile nelle pagine risultato;
* opzioni avanzate solo in modal;
* default ragionevoli;
* messaggi chiari in caso di scenario stale o invalid;
* filename leggibile;
* conferma dopo download;
* nessun linguaggio tecnico eccessivo;
* warning chiaro se i costi non sono disponibili;
* distinzione tra CSV per analisi e JSON per backup/import;
* disclaimer visibile prima dell’export.

### 23. Definizione di completato

Il FP-014 può essere considerato completo quando:

* l’utente può esportare scenario line-item CSV;
* l’utente può esportare scenario summary CSV;
* l’utente può esportare segment breakdown CSV;
* l’utente può esportare task breakdown CSV, se disponibile;
* l’utente può esportare model breakdown CSV;
* l’utente può esportare funding breakdown CSV;
* l’utente può esportare comparison CSV;
* l’utente può esportare portfolio CSV;
* i CSV includono valori min/mid/max;
* i CSV includono metadata scenario;
* i CSV includono metadata assumption pack;
* i CSV includono metadata funding;
* i CSV gestiscono valori N/A;
* i CSV gestiscono scenario stale;
* i CSV non presentano scenario invalid come valido;
* i CSV usano encoding UTF-8;
* i valori con caratteri speciali sono escaped;
* i filename sono leggibili e sicuri;
* l’export non modifica dati scenario;
* l’export aggiorna export history;
* la UI mostra conferma o errore;
* il CSV include disclaimer o indicazione di stima;
* tutto funziona senza backend e resta compatibile con GitHub Pages.
