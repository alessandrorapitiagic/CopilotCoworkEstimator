## FP-015 — Export e import JSON

---

### 1. Obiettivo funzionale

La funzionalità “Export e import JSON” consente all’utente di esportare e importare dati completi dell’app in formato JSON.

A differenza dell’export CSV, che è pensato per analisi tabellare, il JSON è pensato per:

* backup completo;
* ripristino dati;
* portabilità tra browser o dispositivi;
* condivisione di scenari;
* import di scenario library;
* trasferimento di assumption pack custom;
* conservazione storica di stime;
* migrazione controllata tra versioni dell’app;
* collaborazione senza backend.

Il JSON deve contenere non solo i risultati numerici, ma anche tutti gli oggetti necessari per ricostruire uno scenario:

* company;
* scenario;
* workforce segments;
* usage profiles custom;
* task presets custom;
* models custom;
* assumption packs;
* funding plans;
* calculation results;
* calculation trace;
* warning;
* metadata;
* schema version.

L’obiettivo è permettere all’utente di esportare un insieme di dati, conservarlo, inviarlo a un collega o reimportarlo in futuro mantenendo auditabilità e coerenza.

---

### 2. Attori coinvolti

Gli attori principali sono:

* Consulente;
* Solution architect;
* Sales specialist;
* Account manager;
* Responsabile Finance;
* Responsabile IT;
* Cliente finale;
* Utente che vuole fare backup;
* Utente che cambia browser o dispositivo;
* Utente che riceve uno scenario da un collega;
* Utente che importa una libreria di scenari;
* Utente che lavora senza backend o account.

---

### 3. Prerequisiti

Per esportare o importare JSON devono essere disponibili:

* storage locale;
* schema dati applicativo;
* validation service;
* import service;
* export service;
* migration service;
* ID generator;
* conflict resolver;
* data recovery service;
* default system data;
* gestione errori UI;
* supporto download/upload file nel browser.

L’export JSON deve funzionare anche se l’app è deployata come sito statico.

L’import JSON deve funzionare senza chiamate server.

---

### 4. Trigger

La funzionalità viene attivata quando:

* l’utente clicca “Export JSON” da Data management;
* l’utente esporta un singolo scenario;
* l’utente esporta una company;
* l’utente esporta un portfolio;
* l’utente esporta un assumption pack custom;
* l’utente esporta una scenario library;
* l’utente clicca “Import JSON”;
* l’utente carica un backup;
* l’utente apre un file JSON ricevuto da un collega;
* l’utente importa uno scenario condiviso;
* l’utente deve ripristinare dati dopo reset;
* l’utente cambia browser o dispositivo;
* l’utente vuole archiviare una stima storica;
* l’utente vuole migrare dati da una versione precedente dell’app.

---

### 5. Descrizione funzionale dettagliata

Il sistema deve consentire export e import JSON in modo controllato e sicuro.

L’utente deve poter esportare diversi livelli di dati:

1. **Full backup**

   * tutto lo stato locale dell’app.

2. **Company export**

   * una company con tutti i suoi scenari e dipendenze.

3. **Scenario export**

   * un singolo scenario con company minima e dipendenze necessarie.

4. **Scenario library export**

   * più scenari selezionati.

5. **Assumption pack export**

   * uno o più assumption pack custom.

6. **Custom assets export**

   * profili, task, modelli e pack custom.

7. **Draft export**

   * bozza wizard non ancora salvata.

Il sistema deve anche supportare import con modalità diverse:

* import as new;
* merge;
* replace all;
* import selected;
* review before import.

Per MVP sono sufficienti:

* full backup export;
* scenario export;
* import as new;
* replace all con conferma forte;
* import preview;
* validazione schema;
* gestione conflitti ID.

L’import non deve mai sovrascrivere dati locali senza conferma esplicita.

### 6. Tipologie di export JSON

## 6.1 Full backup JSON

Contiene l’intero stato locale dell’app.

Include:

* companies;
* scenarios;
* segments;
* profiles;
* tasks;
* models;
* assumption packs;
* funding plans;
* calculation results;
* comparisons, se salvate;
* preferences;
* metadata;
* schema version.

Serve per:

* backup;
* migrazione dispositivo;
* ripristino;
* archiviazione completa.

---

## 6.2 Company JSON

Contiene una company e tutte le sue dipendenze.

Include:

* company;
* scenari della company;
* segmenti company baseline;
* segmenti scenario-specific;
* funding plans;
* calculation results;
* assumption packs custom usati;
* custom profiles usati;
* custom tasks usati;
* custom models usati.

Serve per:

* inviare un cliente a un collega;
* archiviare una stima cliente;
* esportare un workspace aziendale.

---

## 6.3 Scenario JSON

Contiene un singolo scenario e tutte le dipendenze necessarie per ricalcolarlo o visualizzarlo.

Include:

* scenario;
* company minima;
* segmenti dello scenario;
* funding plan;
* calculation result;
* calculation trace;
* assumption snapshot;
* assumption pack usato, se custom;
* custom profile usati;
* custom task usati;
* custom model usati;
* warning.

Serve per:

* condividere uno scenario;
* archiviare una singola stima;
* importare una simulazione in un altro browser.

---

## 6.4 Scenario library JSON

Contiene più scenari selezionati.

Può includere scenari di una o più company.

Serve per:

* passare una libreria di stime;
* creare template di scenari;
* trasferire un set di simulazioni.

---

## 6.5 Assumption pack JSON

Contiene uno o più assumption pack.

Serve per:

* condividere ipotesi di calcolo;
* standardizzare pack tra colleghi;
* conservare versioni custom.

---

## 6.6 Draft JSON

Contiene una bozza non ancora salvata.

Serve per:

* proteggere lavoro incompleto;
* inviare una bozza a un collega;
* recuperare lavoro se localStorage non è disponibile.

### 9. Regole di business

#### BR-FP015-001 — JSON come formato di portabilità

Il JSON è il formato principale per backup, ripristino e portabilità.

Non deve essere sostituito dal CSV.

---

#### BR-FP015-002 — Export browser-only

L’export JSON deve essere generato interamente nel browser.

Non deve richiedere backend.

---

#### BR-FP015-003 — Import browser-only

L’import JSON deve essere eseguito interamente nel browser.

Il file deve essere letto localmente e non inviato a server.

---

#### BR-FP015-004 — Schema version obbligatoria

Ogni JSON esportato deve contenere `schemaVersion`.

Se un JSON importato non contiene schema version, deve essere trattato come legacy.

---

#### BR-FP015-005 — Export type obbligatorio

Ogni payload JSON deve indicare `exportType`.

Questo permette all’import di sapere come interpretare il contenuto.

---

#### BR-FP015-006 — Metadata obbligatori

Ogni export deve contenere metadata minimi:

* exportedAt;
* sourceApp;
* counts;
* disclaimer;
* schemaVersion.

---

#### BR-FP015-007 — Dipendenze incluse

Uno scenario esportato deve includere tutte le dipendenze necessarie a visualizzarlo e ricalcolarlo.

Esempi:

* funding plan;
* assumption pack custom;
* custom usage profile;
* custom task preset;
* custom model;
* calculation result.

---

#### BR-FP015-008 — System data non duplicati inutilmente

I dati system standard possono non essere inclusi interamente se sono ricostruibili dall’app.

Tuttavia, se uno scenario dipende da una versione specifica, deve essere incluso almeno lo snapshot o riferimento sufficiente.

---

#### BR-FP015-009 — Assumption snapshot obbligatorio per risultati

Se si esporta un calculation result, deve essere incluso anche:

* assumption pack ID;
* assumption pack version;
* source date;
* assumption snapshot;
* calculation trace.

---

#### BR-FP015-010 — Import preview obbligatoria

Prima di importare, il sistema deve mostrare una preview.

La preview deve indicare:

* tipo file;
* numero company;
* numero scenari;
* pack custom;
* profili custom;
* warning;
* conflitti;
* modalità import disponibili.

---

#### BR-FP015-011 — Nessuna sovrascrittura senza conferma

L’import non deve sovrascrivere dati locali senza conferma esplicita.

---

#### BR-FP015-012 — Import as new default

La modalità predefinita di import deve essere `importAsNew`.

Questa modalità genera nuovi ID per evitare conflitti.

---

#### BR-FP015-013 — Replace all con conferma forte

La modalità `replaceAll` deve richiedere conferma forte.

Esempio:

```text id="p467n0"
Type REPLACE to overwrite local data.
```

---

#### BR-FP015-014 — Merge avanzato

Il merge può essere previsto in versione avanzata.

Per MVP, se il merge è troppo complesso, usare import as new.

---

#### BR-FP015-015 — Conflitti ID

Se un oggetto importato ha ID già presente, il sistema deve:

* generare nuovo ID;
* aggiornare riferimenti interni;
* mostrare mapping nella preview o nel risultato.

---

#### BR-FP015-016 — Riferimenti interni coerenti

Durante import, se un scenario riceve nuovo ID, tutti gli oggetti collegati devono essere aggiornati:

* segments;
* funding plan;
* calculation result;
* comparison;
* warning;
* trace.

---

#### BR-FP015-017 — Company duplicate

Se una company importata ha stesso nome di una company locale, il sistema deve mostrare warning.

Non è bloccante.

---

#### BR-FP015-018 — Scenario duplicate

Se uno scenario importato ha stesso nome di uno scenario locale, il sistema deve mostrare warning.

Non è bloccante se import as new.

---

#### BR-FP015-019 — Assumption pack duplicate

Se un assumption pack importato coincide con uno locale per ID o nome/versione, il sistema deve proporre:

* reuse existing;
* import as new;
* skip;
* replace, solo con conferma.

Per MVP: import as new o reuse exact match.

---

#### BR-FP015-020 — Schema migration

Se il JSON importato usa schema vecchio, il sistema deve tentare migrazione.

Se la migrazione fallisce, l’import viene bloccato.

---

#### BR-FP015-021 — Schema futuro

Se il JSON usa schema più recente della versione supportata, il sistema deve mostrare warning o bloccare import.

Per sicurezza, bloccare import se incompatibile.

---

#### BR-FP015-022 — Dati corrotti

Se il JSON non è parsabile, il sistema deve mostrare errore e non modificare dati locali.

---

#### BR-FP015-023 — Import parziale

Se alcuni oggetti sono validi e altri no, il sistema può proporre import parziale.

Per MVP si consiglia:

* import tutto se valido;
* blocca se errori bloccanti;
* warning per oggetti opzionali non validi.

---

#### BR-FP015-024 — Dati demo

Se il JSON contiene dati demo, il sistema deve indicarlo.

---

#### BR-FP015-025 — Dati shared

Se il JSON deriva da link condiviso o scenario condiviso, deve essere marcato come imported/shared.

---

#### BR-FP015-026 — Import non salva automaticamente fino a conferma

La preview di import non deve modificare dati locali.

I dati vengono salvati solo dopo conferma utente.

---

#### BR-FP015-027 — Export non modifica dati

L’export non deve modificare gli oggetti esportati.

Può aggiornare solo metadata locali come lastExportAt.

---

#### BR-FP015-028 — Checksum opzionale

Il payload può includere checksum per verificare integrità.

Per MVP è opzionale.

---

#### BR-FP015-029 — Compressione opzionale

Per file JSON non serve compressione.

Per share link si può usare JSON compresso nel link, ma questa logica appartiene principalmente a FP-016.

---

#### BR-FP015-030 — Privacy locale

La UI deve chiarire che il file JSON può contenere nomi company, note e dati di scenario inseriti dall’utente.

---

### 12. Flusso principale — Export full backup

1. L’utente apre Data management.
2. Clicca “Export full backup”.
3. Il sistema raccoglie l’intero stato locale.
4. Valida lo stato.
5. Costruisce payload JSON.
6. Include metadata e disclaimer.
7. Genera file JSON.
8. Il browser scarica il file.
9. Il sistema aggiorna `lastExportAt`.
10. La UI mostra conferma.

---

### 13. Flusso principale — Export scenario

1. L’utente apre uno scenario.
2. Clicca “Export scenario JSON”.
3. Il sistema raccoglie scenario e dipendenze.
4. Include calculation result e assumption snapshot.
5. Valida payload.
6. Genera file JSON.
7. Il browser scarica il file.
8. La UI mostra conferma.

---

### 14. Flusso principale — Import JSON

1. L’utente clicca “Import JSON”.
2. Seleziona file.
3. Il sistema legge il file.
4. Esegue parsing JSON.
5. Valida schema.
6. Esegue eventuale migrazione.
7. Rileva conflitti.
8. Mostra import preview.
9. L’utente sceglie modalità import.
10. L’utente conferma.
11. Il sistema applica import.
12. Salva dati localmente.
13. Mostra riepilogo import.

---

### 15. Flussi alternativi

#### A1 — File non JSON

1. L’utente seleziona un file non JSON.
2. Il sistema prova a leggerlo.
3. Il parsing fallisce.
4. Il sistema mostra errore.
5. Nessun dato locale viene modificato.

---

#### A2 — JSON valido ma non compatibile

1. Il file è JSON valido.
2. Non contiene struttura attesa.
3. Il sistema mostra messaggio:
   “This JSON is not a valid Cowork estimator export.”
4. Import bloccato.

---

#### A3 — Schema vecchio

1. Il file contiene schemaVersion precedente.
2. Il sistema tenta migrazione.
3. Se riesce, mostra warning.
4. L’utente può importare.
5. Se fallisce, import bloccato.

---

#### A4 — Schema futuro

1. Il file contiene schemaVersion superiore a quella supportata.
2. Il sistema mostra warning.
3. Import bloccato oppure consentito solo se compatibile.
4. Per MVP: bloccare.

---

#### A5 — ID duplicati

1. Il file contiene ID già presenti localmente.
2. Il sistema rileva conflitti.
3. In import as new genera nuovi ID.
4. Aggiorna riferimenti interni.
5. Mostra mapping o riepilogo.

---

#### A6 — Company duplicata per nome

1. Il file contiene company con nome già presente.
2. Il sistema mostra warning.
3. Import as new resta consentito.
4. Il nome può essere suffissato con “Imported”.

---

#### A7 — Assumption pack già presente

1. Il file contiene un pack già presente.
2. Il sistema rileva corrispondenza.
3. Può riusare il pack locale.
4. Se non è identico, importa come nuovo.
5. Mostra warning.

---

#### A8 — Scenario senza company

1. Il file contiene scenario ma manca company collegata.
2. Il sistema può:

   * bloccare import;
   * creare company placeholder;
   * chiedere mapping.
3. Per MVP: bloccare o creare placeholder con warning.

---

#### A9 — Import parziale con warning

1. Il file contiene alcuni oggetti opzionali non validi.
2. Gli oggetti principali sono validi.
3. Il sistema mostra warning.
4. L’utente può importare solo dati validi, se supportato.
5. Il risultato import indica cosa è stato escluso.

---

#### A10 — Replace all annullato

1. L’utente sceglie replace all.
2. Il sistema richiede conferma forte.
3. L’utente annulla.
4. Nessun dato locale viene modificato.

---

#### A11 — Import con localStorage pieno

1. L’utente conferma import.
2. Il salvataggio fallisce per storage pieno.
3. Il sistema mostra errore.
4. I dati locali precedenti restano intatti.
5. L’utente può esportare backup o liberare spazio.

---

#### A12 — Import scenario condiviso

1. L’utente importa uno scenario ricevuto.
2. Il sistema lo marca come imported/shared.
3. Genera nuovi ID.
4. Mostra “Needs review” se contiene assumption custom o fallback.
5. L’utente può salvarlo localmente.

---

### 16. Validazioni

#### Validazioni bloccanti

* file mancante;
* file non leggibile;
* JSON non parsabile;
* exportType mancante;
* schemaVersion mancante e migration impossibile;
* struttura root non valida;
* data object mancante;
* scenario senza ID;
* scenario senza companyId;
* company senza ID;
* ID duplicati interni al payload;
* riferimenti interni non risolvibili;
* assumption pack mancante e nessuno snapshot disponibile;
* funding plan mancante per scenario reviewed;
* calculation result invalido;
* schema futuro incompatibile;
* replace all non confermato;
* errore salvataggio finale.

---

#### Warning non bloccanti

* schema vecchio migrato;
* dati importati con nuovi ID;
* company name duplicato;
* scenario name duplicato;
* assumption pack duplicato;
* custom assumptions presenti;
* calculation result stale;
* scenario draft;
* scenario archived;
* modello mancante mappato ad Auto;
* funding incompleto sostituito con PAYG default;
* dati demo presenti;
* dati shared presenti;
* export include dati sensibili inseriti dall’utente;
* file molto grande;
* import parziale;
* localStorage quasi pieno;
* pack deprecated;
* scenario con warning critici.

### 22. Criteri di accettazione

#### AC-001 — Export full backup

Given esistono dati locali,
when l’utente clicca Export full backup,
then il sistema genera un JSON contenente lo stato completo dell’app.

---

#### AC-002 — Export scenario

Given esiste uno scenario salvato,
when l’utente esporta scenario JSON,
then il file contiene scenario, company, segmenti, funding plan, calculation result e dipendenze necessarie.

---

#### AC-003 — Export assumption pack

Given esiste un assumption pack custom,
when l’utente esporta assumption pack JSON,
then il file contiene il pack con versione, source date e valori custom.

---

#### AC-004 — Metadata export

Given viene generato un JSON,
when il file viene creato,
then contiene exportId, exportType, schemaVersion, exportedAt, sourceApp e metadata.

---

#### AC-005 — Disclaimer export

Given viene esportato un JSON,
when il file viene generato,
then contiene una nota che indica che le stime sono basate su assunzioni.

---

#### AC-006 — Import JSON valido

Given l’utente seleziona un JSON valido,
when il sistema lo legge,
then mostra import preview prima di modificare dati locali.

---

#### AC-007 — Preview import

Given il file contiene company e scenari,
when viene mostrata la preview,
then l’utente vede numero company, numero scenari, custom assumptions e warning.

---

#### AC-008 — Import as new

Given l’utente seleziona import as new,
when conferma,
then il sistema genera nuovi ID e aggiunge i dati senza sovrascrivere quelli locali.

---

#### AC-009 — Remap riferimenti

Given gli oggetti importati ricevono nuovi ID,
when l’import termina,
then i riferimenti interni tra scenario, segmenti, funding e calculation result restano coerenti.

---

#### AC-010 — Replace all

Given l’utente seleziona replace all,
when conferma con conferma forte,
then i dati locali vengono sostituiti dal payload importato e i system data vengono garantiti.

---

#### AC-011 — Replace all annullato

Given l’utente seleziona replace all,
when non conferma correttamente,
then nessun dato locale viene modificato.

---

#### AC-012 — File non JSON

Given l’utente seleziona un file non JSON,
when il sistema tenta import,
then mostra errore e non modifica dati locali.

---

#### AC-013 — JSON non compatibile

Given il file è JSON valido ma non ha struttura dell’app,
when il sistema valida,
then blocca import.

---

#### AC-014 — Schema vecchio migrabile

Given il file usa schema vecchio supportato,
when viene importato,
then il sistema migra i dati e mostra warning.

---

#### AC-015 — Schema futuro

Given il file usa schema più recente non supportato,
when l’utente prova a importarlo,
then il sistema blocca import.

---

#### AC-016 — ID duplicati

Given il payload contiene ID già presenti localmente,
when import as new viene applicato,
then il sistema genera nuovi ID.

---

#### AC-017 — Company duplicata

Given una company importata ha stesso nome di una locale,
when viene importata come nuova,
then il sistema mostra warning ma consente import.

---

#### AC-018 — Scenario senza company

Given uno scenario importato non ha company risolvibile,
when il sistema valida,
then blocca import o crea placeholder con warning secondo configurazione MVP.

---

#### AC-019 — Assumption pack custom incluso

Given uno scenario esportato usa assumption pack custom,
when viene esportato JSON,
then il pack custom viene incluso.

---

#### AC-020 — Import con funding incompleto

Given uno scenario importato ha funding plan incompleto,
when il sistema importa,
then lo scenario viene marcato Needs review o usa PAYG default con warning.

---

#### AC-021 — localStorage pieno

Given lo storage è pieno,
when l’utente conferma import,
then il sistema mostra errore e non corrompe dati locali.

---

#### AC-022 — Export non modifica dati

Given l’utente esporta JSON,
when il file viene generato,
then gli oggetti esportati non vengono modificati.

---

#### AC-023 — Import result summary

Given l’import termina,
when il sistema mostra risultato,
then indica quanti oggetti sono stati importati e quali warning sono presenti.

---

#### AC-024 — Privacy notice

Given l’utente esporta o importa JSON,
when vede la UI relativa,
then il sistema avvisa che il file può contenere dati inseriti dall’utente.

---

#### AC-025 — Funzionamento senza backend

Given l’app è servita come sito statico,
when l’utente esporta o importa JSON,
then tutto avviene nel browser senza chiamate server.

### 23. Priorità

Priorità funzionale: **Must have**

L’export/import JSON è fondamentale perché rende sostenibile l’app statica.

Senza JSON:

* l’utente non può fare backup affidabile;
* i dati restano bloccati nel browser;
* non è possibile trasferire scenari tra dispositivi;
* non è possibile condividere una scenario library;
* non è possibile ripristinare dopo reset o cache clear;
* non è possibile portare assumption pack custom tra utenti.

Con JSON, l’app diventa un vero workbench portabile e non solo uno strumento locale fragile.

La UX deve essere chiara e protettiva.

Principi UX:

* spiegare che JSON serve per backup e import;

* usare import as new come default;
* evidenziare custom assumptions;
* evidenziare dati demo;
* mostrare warning schema e compatibilità;
* mostrare riepilogo import;
* rendere semplice export full backup;
* rendere semplice export scenario;
* avvisare che il file può contenere nomi company e note;
* mantenere linguaggio non tecnico dove possibile.
