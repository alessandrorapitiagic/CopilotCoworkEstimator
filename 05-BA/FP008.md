## FP-008 — Assumption pack versionati

---

### 1. Obiettivo funzionale

La funzionalità “Assumption pack versionati” consente di gestire insiemi strutturati, versionati e auditabili di assunzioni utilizzate dal motore di calcolo.

Un assumption pack rappresenta la base logica con cui l’app stima:

* crediti per task light;
* crediti per task medium;
* crediti per task heavy;
* model factor;
* context factor;
* tools factor;
* runtime factor;
* browser factor;
* image factor;
* prezzo per credito;
* sconti o funding assumptions;
* soglie budget;
* note e fonti;
* data di validità;
* disclaimer.

Questa funzionalità è essenziale perché il modello pubblico di pricing Cowork non è interamente deterministico. L’app deve quindi evitare di nascondere le assunzioni dentro il codice o dentro formule implicite.

Ogni scenario deve sempre sapere:

* quale assumption pack ha usato;
* quale versione del pack;
* quando il pack è stato creato o aggiornato;
* quali valori sono standard;
* quali valori sono custom;
* quali fonti supportano le assunzioni;
* se il pack è obsoleto;
* se il pack è stato modificato dall’utente.

L’obiettivo è rendere ogni stima spiegabile, confrontabile e ricostruibile nel tempo.

---

### 2. Attori coinvolti

Gli attori principali sono:

* Consulente;
* Solution architect;
* Sales specialist;
* Account manager;
* Responsabile Finance;
* Responsabile IT;
* Utente avanzato;
* Utente che aggiorna le assunzioni;
* Utente che confronta stime storiche;
* Cliente finale che vuole capire come è stata calcolata la stima.

---

### 3. Prerequisiti

Per usare gli assumption pack devono essere disponibili:

* catalogo iniziale di assumption pack di sistema;
* credit bands light/medium/heavy;
* catalogo modelli AI;
* task preset;
* profili di utilizzo;
* funding plan;
* motore di calcolo;
* servizio di validazione;
* servizio di persistenza locale;
* export/import JSON.

La funzionalità deve essere disponibile anche al primo accesso, perché almeno un assumption pack di sistema deve essere inizializzato automaticamente.

---

### 4. Trigger

La funzionalità viene attivata quando:

* l’utente apre la pagina “Assumptions”;
* l’utente seleziona un assumption pack in un wizard;
* l’utente crea un nuovo scenario;
* l’utente duplica uno scenario esistente;
* l’utente importa uno scenario con assumption pack custom;
* l’utente apre un link condiviso con assumption pack incluso;
* l’utente modifica un model factor;
* l’utente modifica credit bands;
* l’utente aggiorna il prezzo per credito;
* l’utente confronta due scenari con pack diversi;
* il sistema rileva che uno scenario usa un pack obsoleto;
* il sistema ricalcola uno scenario;
* l’utente esporta o importa dati JSON.

---

### 5. Descrizione funzionale dettagliata

Il sistema deve gestire gli assumption pack come entità autonome e versionate.

Ogni assumption pack deve contenere tutti i parametri necessari al motore di calcolo per trasformare dati aziendali e comportamentali in crediti e costi stimati.

L’app deve includere almeno un assumption pack di sistema, ad esempio:

* Current public range assumptions;
* Legacy / Frontier-style assumptions, se mantenuto per confronto storico;
* Custom assumptions create dall’utente.

I pack di sistema devono essere protetti:

* visualizzabili;
* selezionabili;
* duplicabili;
* non modificabili direttamente;
* non eliminabili.

I pack custom devono essere:

* modificabili;
* duplicabili;
* eliminabili se non in uso;
* esportabili;
* importabili;
* marcati come custom nei report;
* associabili a scenari.

Ogni scenario deve referenziare un assumption pack tramite ID e versione.

Se il pack cambia dopo la creazione dello scenario, l’app deve evitare ambiguità. Lo scenario deve poter mantenere la versione storica oppure essere duplicato/aggiornato con la nuova versione.

La funzionalità deve quindi supportare sia l’evoluzione delle assunzioni sia la conservazione delle stime storiche.

### 7. Struttura credit bands

L’assumption pack deve contenere le bande di consumo per intensità.

Esempio struttura:

```text id="iuxod4"
creditBands {
  light: {
    min: number
    mid: number
    max: number
    description: string
    examples: string[]
  }
  medium: {
    min: number
    mid: number
    max: number
    description: string
    examples: string[]
  }
  heavy: {
    min: number
    mid: number
    max: number
    description: string
    examples: string[]
    isOpenEnded: boolean
  }
}
```

Regola fondamentale:

```text id="lv0gh4"
min <= mid <= max
```

Per i task heavy, se la fonte indica solo una soglia minima, il pack deve comunque contenere midpoint e massimo configurabili per consentire calcolo range-based.

---

### 8. Struttura model factors

L’assumption pack deve definire i factor applicati ai modelli.

Esempio:

```text id="2e2e0e"
modelFactors {
  auto: {
    factor: number
    label: string
    notes: string
  }
  claudeSonnet46: {
    factor: number
    label: string
    notes: string
  }
  claudeOpus48: {
    factor: number
    label: string
    notes: string
  }
  sonnetOpusAdvisor: {
    factor: number
    label: string
    notes: string
  }
  gpt55Frontier: {
    factor: number
    label: string
    notes: string
  }
  imagen2: {
    factor: number
    label: string
    notes: string
  }
}
```

I factor devono essere considerati assunzioni relative, non prezzi ufficiali.

### 10. Tipologie di assumption pack

#### 10.1 System current range pack

Pack di sistema principale.

Caratteristiche:

* usa bande min/mid/max;
* rappresenta l’assunzione più aggiornata;
* non è modificabile direttamente;
* è il default per nuovi scenari;
* può essere duplicato.

---

#### 10.2 Legacy pack

Pack storico opzionale.

Caratteristiche:

* mantiene assunzioni precedenti;
* utile per confronto;
* marcato come legacy;
* non è default;
* può essere selezionato consapevolmente.

---

#### 10.3 Point estimate pack

Pack basato su valori puntuali.

Caratteristiche:

* min = mid = max oppure usa solo mid;
* utile per stime rapide;
* deve essere marcato come point estimate;
* meno prudente del range-based.

---

#### 10.4 Custom pack

Pack creato dall’utente.

Caratteristiche:

* modificabile;
* esportabile;
* importabile;
* marcato come custom;
* può derivare da pack system o da zero;
* deve indicare chiaramente valori modificati.

---

#### 10.5 Imported pack

Pack importato da JSON o link.

Caratteristiche:

* marcato come imported;
* validato;
* può richiedere review;
* può contenere valori custom;
* può essere convertito in custom locale.

---

### 11. Regole di business

#### BR-FP008-001 — Ogni scenario deve avere un assumption pack

Uno scenario non può essere calcolato senza assumption pack.

Se il pack manca, il sistema deve mostrare errore bloccante.

---

#### BR-FP008-002 — Pack default obbligatorio

L’app deve avere sempre un assumption pack default corrente.

Se il pack default non è presente, il sistema deve ricrearlo dai dati di sistema.

---

#### BR-FP008-003 — Pack di sistema non modificabili direttamente

Gli assumption pack di sistema non devono essere modificabili direttamente.

L’utente deve duplicarli per personalizzarli.

---

#### BR-FP008-004 — Pack di sistema non eliminabili

Gli assumption pack di sistema non possono essere eliminati.

---

#### BR-FP008-005 — Nome pack obbligatorio

Ogni assumption pack deve avere un nome.

Non è possibile salvare un pack custom senza nome.

---

#### BR-FP008-006 — Versione obbligatoria

Ogni assumption pack deve avere una versione.

La versione può essere generata automaticamente.

---

#### BR-FP008-007 — Source date obbligatoria

Ogni assumption pack deve avere una `sourceDate`.

Serve per auditabilità e per rilevare pack obsoleti.

---

#### BR-FP008-008 — Credit bands coerenti

Per ogni intensità deve valere:

```text id="a2axnx"
min <= mid <= max
```

Valori incoerenti sono bloccanti.

---

#### BR-FP008-009 — Valori credito non negativi

Tutti i valori di credito devono essere maggiori o uguali a zero.

Valori negativi sono bloccanti.

---

#### BR-FP008-010 — Heavy open-ended

Se una banda heavy è open-ended, l’app deve comunque richiedere un valore massimo tecnico per la stima.

Il valore massimo deve essere marcato come assunzione.

---

#### BR-FP008-011 — Factor non negativi

Tutti i factor devono essere maggiori o uguali a zero.

Valori negativi sono bloccanti.

---

#### BR-FP008-012 — Factor elevati

Factor superiori alla soglia configurata devono generare warning.

Esempio soglia iniziale:

```text id="p7ua2w"
factor > 3
```

---

#### BR-FP008-013 — PAYG price default

L’assumption pack deve contenere un prezzo per credito di default.

Questo valore viene usato dal funding plan PAYG iniziale.

---

#### BR-FP008-014 — Prezzo credito modificabile

Il prezzo per credito può essere modificato solo:

* nel funding plan dello scenario;
* oppure in un assumption pack custom.

---

#### BR-FP008-015 — Modifica pack in uso

Se un pack custom è usato da scenari, modificarlo deve generare una scelta:

1. aggiornare il pack e ricalcolare gli scenari collegati;
2. creare una nuova versione;
3. duplicare il pack;
4. annullare.

Per MVP si consiglia: creare automaticamente una nuova versione o chiedere conferma.

---

#### BR-FP008-016 — Scenario storico

Uno scenario salvato deve mantenere riferimento alla versione del pack usata al momento del calcolo.

Se il pack viene aggiornato, lo scenario deve mostrare:

* “calcolato con versione precedente”;
* opzione “ricalcola con nuova versione”;
* opzione “duplica scenario con nuova versione”.

---

#### BR-FP008-017 — Deprecazione pack

Un pack può essere marcato come deprecated.

Un pack deprecated:

* resta disponibile per scenari storici;
* non è proposto come default;
* genera warning se usato in nuovo scenario;
* può essere selezionato solo consapevolmente.

---

#### BR-FP008-018 — Custom pack nel report

Se uno scenario usa un custom pack, il report deve indicarlo chiaramente.

---

#### BR-FP008-019 — Imported pack

Un pack importato deve essere validato.

Se valido, può essere salvato come imported/custom.

Se non valido, deve essere rifiutato o richiedere correzione.

---

#### BR-FP008-020 — Export pack

Un assumption pack custom deve essere incluso nell’export JSON se usato da uno scenario esportato.

---

#### BR-FP008-021 — Import con conflitto ID

Se un pack importato ha ID già presente, il sistema deve generare un nuovo ID oppure chiedere merge.

Per MVP: generare nuovo ID e mantenere il nome con suffisso “Imported”.

---

#### BR-FP008-022 — Pack e modelli mancanti

Se un pack importato contiene factor per modelli non presenti nel catalogo locale, il sistema deve:

* conservare i factor;
* mostrare warning;
* proporre mapping;
* usare Auto come fallback nei calcoli se il modello non è disponibile.

---

#### BR-FP008-023 — Pack e task preset

Se un task preset fa riferimento a un pack non disponibile, il sistema deve usare il pack default e mostrare warning.

---

#### BR-FP008-024 — Auditabilità obbligatoria

Ogni calcolo deve poter mostrare:

* assumption pack name;
* version;
* source date;
* source type;
* valori usati;
* custom changes;
* disclaimer.

---

#### BR-FP008-025 — Confronto pack

L’app deve permettere di confrontare due assumption pack.

Il confronto deve mostrare differenze su:

* credit bands;
* model factors;
* context factors;
* tools factors;
* runtime factors;
* browser factors;
* image factors;
* funding defaults.

---

### 12. Logiche di calcolo

L’assumption pack alimenta il motore di calcolo.

Recupero credit band:

```text id="4fgged"
creditBand =
  assumptionPack.creditBands[taskIntensity]
```

Calcolo range per task:

```text id="9hbjbo"
taskCreditsMin = creditBand.min
taskCreditsMid = creditBand.mid
taskCreditsMax = creditBand.max
```

Recupero model factor:

```text id="aapbwh"
modelFactor =
  assumptionPack.modelFactors[modelId]?.factor
  ?? 1
```

Recupero driver factor:

```text id="kus8lc"
contextFactor =
  assumptionPack.contextFactors[contextLevel]
  ?? 1

toolsFactor =
  assumptionPack.toolsFactors[toolsLevel]
  ?? 1

runtimeFactor =
  assumptionPack.runtimeFactors[runtimeLevel]
  ?? 1

browserFactor =
  assumptionPack.browserFactors[browserLevel]
  ?? 1

imageFactor =
  assumptionPack.imageFactors[imageLevel]
  ?? 1
```

Formula finale:

```text id="c5szhc"
adjustedCredits =
  baseCredits
  × modelFactor
  × contextFactor
  × toolsFactor
  × runtimeFactor
  × browserFactor
  × imageFactor
```

Determinazione pack obsoleto:

```text id="ytbbas"
isOutdated =
  scenario.assumptionPackId == pack.id
  && scenario.assumptionPackVersion != pack.version
```

Determinazione pack deprecated:

```text id="ntohcy"
isDeprecated =
  assumptionPack.isDeprecated == true
```

Determinazione pack custom:

```text id="1195ap"
isCustom =
  assumptionPack.sourceType == "custom"
  || assumptionPack.isSystemDefault == false
```

### 13. Flusso principale — Visualizzazione assumption pack

1. L’utente apre la pagina “Assumptions”.
2. Il sistema carica tutti gli assumption pack.
3. Il sistema mostra:

   * pack di sistema;
   * pack custom;
   * pack importati;
   * pack deprecated.
4. Ogni pack mostra:

   * nome;
   * versione;
   * source date;
   * stato;
   * numero scenari collegati;
   * default sì/no;
   * warning.
5. L’utente apre il dettaglio di un pack.
6. Il sistema mostra credit bands, factor, funding defaults e disclaimer.
7. Se il pack è system, l’utente può duplicarlo.
8. Se il pack è custom, l’utente può modificarlo, duplicarlo o eliminarlo se non in uso.

---

### 14. Flusso principale — Creazione pack custom da duplicazione

1. L’utente apre un pack di sistema.
2. Clicca “Duplicate”.
3. Il sistema crea una copia custom.
4. Il nome viene impostato come “Copy of [pack name]”.
5. La versione viene impostata a `1.0.0` o incrementata.
6. L’utente modifica valori desiderati.
7. Il sistema valida.
8. L’utente salva.
9. Il pack viene salvato in localStorage.
10. Il pack diventa selezionabile nei nuovi scenari.

---

### 15. Flusso principale — Selezione pack in scenario

1. L’utente entra nel wizard scenario.
2. Arriva allo step “Models & assumptions”.
3. Il sistema propone il pack default corrente.
4. L’utente può scegliere un altro pack.
5. Il sistema mostra riepilogo:

   * nome;
   * versione;
   * source date;
   * tipo;
   * eventuali warning.
6. Il sistema ricalcola la preview.
7. Lo scenario salva riferimento a pack ID e versione.

---

### 16. Flussi alternativi

#### A1 — Modifica pack custom non in uso

1. L’utente apre un pack custom non usato.
2. Modifica credit bands o factor.
3. Il sistema valida i valori.
4. L’utente salva.
5. Il pack viene aggiornato.
6. `updatedAt` viene aggiornato.

---

#### A2 — Modifica pack custom in uso

1. L’utente apre un pack custom usato da scenari.
2. Modifica un valore.
3. Il sistema mostra warning:
   “Questo pack è usato da scenari esistenti.”
4. L’utente sceglie:

   * crea nuova versione;
   * aggiorna pack esistente;
   * annulla.
5. Se crea nuova versione, gli scenari esistenti restano sulla versione precedente.
6. Se aggiorna, gli scenari collegati vengono marcati da ricalcolare.

---

#### A3 — Pack obsoleto in scenario

1. L’utente apre uno scenario.
2. Il sistema rileva che il pack usato non è più corrente.
3. Mostra warning.
4. L’utente può:

   * mantenere stima storica;
   * ricalcolare con versione corrente;
   * duplicare scenario con nuovo pack.

---

#### A4 — Pack deprecated in nuovo scenario

1. L’utente crea un nuovo scenario.
2. Seleziona un pack deprecated.
3. Il sistema mostra warning.
4. L’utente deve confermare esplicitamente.
5. Il pack viene applicato.

---

#### A5 — Eliminazione pack custom non usato

1. L’utente apre un pack custom non usato.
2. Clicca Delete.
3. Il sistema chiede conferma.
4. L’utente conferma.
5. Il pack viene eliminato.

---

#### A6 — Eliminazione pack custom in uso

1. L’utente tenta di eliminare un pack usato da scenari.
2. Il sistema mostra warning.
3. Per MVP impedisce eliminazione.
4. L’utente deve prima migrare o archiviare gli scenari collegati.

---

#### A7 — Import pack valido

1. L’utente importa JSON.
2. Il payload contiene assumption pack.
3. Il sistema valida struttura e valori.
4. Il pack viene importato come custom/imported.
5. Il sistema mostra riepilogo import.
6. Il pack diventa selezionabile.

---

#### A8 — Import pack con errori

1. L’utente importa JSON.
2. Il pack contiene valori incoerenti.
3. Il sistema mostra lista errori.
4. Il pack non viene importato.
5. I dati esistenti non vengono modificati.

---

#### A9 — Conflitto ID pack importato

1. L’utente importa un pack con ID già presente.
2. Il sistema genera nuovo ID.
3. Aggiunge suffisso al nome o mantiene riferimento source.
4. Mostra warning informativo.

---

#### A10 — Pack con modello non presente

1. Il pack contiene model factor per un modello non presente.
2. Il sistema conserva i dati.
3. Mostra warning.
4. I factor non usati restano disponibili per export o mapping futuro.

---

#### A11 — Set default pack

1. L’utente seleziona un pack.
2. Clicca “Set as default”.
3. Se il pack non è deprecated e valido, diventa default.
4. I nuovi scenari lo useranno automaticamente.
5. Gli scenari esistenti non vengono modificati.

---

#### A12 — Compare packs

1. L’utente seleziona due assumption pack.
2. Clicca “Compare”.
3. Il sistema mostra differenze.
4. L’utente può duplicare uno dei due o applicarlo a uno scenario.

---

### 17. Validazioni

#### Validazioni bloccanti

* nome pack mancante;
* versione mancante;
* source date mancante;
* credit band mancante;
* valore credito negativo;
* min maggiore di mid;
* mid maggiore di max;
* heavy max mancante se necessario per calcolo;
* model factor mancante;
* model factor negativo;
* driver factor negativo;
* funding PAYG price negativo;
* modifica diretta pack system;
* eliminazione pack system;
* eliminazione pack custom in uso;
* import pack con schema non valido;
* scenario senza assumption pack.

---

#### Warning non bloccanti

* pack custom;
* pack imported;
* pack deprecated;
* pack non default;
* source date vecchia;
* factor molto alto;
* factor pari a zero;
* heavy max molto alto;
* pack point estimate;
* pack senza source URL;
* pack senza note;
* pack con modelli non presenti;
* pack usato da molti scenari;
* scenario calcolato con versione precedente;
* pack creato da link condiviso;
* pack con funding defaults custom;
* pack con credit bands molto ampie;
* differenza elevata tra min e max;
* prezzo credito custom;
* pack basato su fonte legacy.

### 23. Criteri di accettazione

#### AC-001 — Pack default disponibile

Given l’app è inizializzata,
when l’utente crea un nuovo scenario,
then il sistema assegna automaticamente l’assumption pack default corrente.

---

#### AC-002 — Scenario senza pack non calcolabile

Given uno scenario non ha assumption pack,
when il sistema prova a calcolarlo,
then mostra errore bloccante.

---

#### AC-003 — Visualizzazione pack

Given l’utente apre la pagina Assumptions,
when i pack sono caricati,
then il sistema mostra nome, versione, source date, stato e numero scenari collegati.

---

#### AC-004 — Pack system non modificabile

Given l’utente apre un pack di sistema,
when prova a modificarlo direttamente,
then il sistema impedisce la modifica e propone duplicazione.

---

#### AC-005 — Pack system non eliminabile

Given l’utente apre un pack di sistema,
when prova a eliminarlo,
then il sistema impedisce l’eliminazione.

---

#### AC-006 — Duplicazione pack system

Given l’utente apre un pack di sistema,
when clicca Duplicate,
then il sistema crea un pack custom con nuovo ID e valori copiati.

---

#### AC-007 — Creazione pack custom valido

Given l’utente crea un pack custom,
when inserisce nome, versione, source date, credit bands e factor validi,
then il sistema salva il pack.

---

#### AC-008 — Nome pack obbligatorio

Given l’utente crea un pack custom,
when lascia vuoto il nome,
then il sistema mostra errore bloccante.

---

#### AC-009 — Versione obbligatoria

Given l’utente crea o modifica un pack,
when la versione è mancante,
then il sistema mostra errore bloccante.

---

#### AC-010 — Source date obbligatoria

Given l’utente crea o modifica un pack,
when la source date è mancante,
then il sistema mostra errore bloccante.

---

#### AC-011 — Credit range valido

Given l’utente modifica credit bands,
when min <= mid <= max,
then il sistema considera il range valido.

---

#### AC-012 — Credit range incoerente

Given l’utente modifica credit bands,
when min > mid oppure mid > max,
then il sistema mostra errore bloccante.

---

#### AC-013 — Crediti negativi non validi

Given l’utente modifica un valore di credito,
when inserisce un numero negativo,
then il sistema mostra errore bloccante.

---

#### AC-014 — Heavy max obbligatorio

Given una banda heavy è usata nel calcolo,
when manca un valore massimo stimato,
then il sistema richiede un valore o usa default configurato con warning.

---

#### AC-015 — Factor negativo non valido

Given l’utente modifica un factor,
when inserisce valore negativo,
then il sistema mostra errore bloccante.

---

#### AC-016 — Factor elevato

Given l’utente imposta un factor superiore alla soglia,
when salva,
then il sistema mostra warning ma consente il salvataggio.

---

#### AC-017 — Pack custom in scenario

Given l’utente seleziona un pack custom in uno scenario,
when apre la review,
then il sistema mostra che lo scenario usa assunzioni personalizzate.

---

#### AC-018 — Scenario salva versione pack

Given l’utente salva uno scenario,
when il salvataggio è completato,
then lo scenario conserva assumptionPackId e assumptionPackVersion.

---

#### AC-019 — Pack aggiornato dopo scenario

Given uno scenario usa una vecchia versione del pack,
when l’utente apre lo scenario,
then il sistema mostra warning “calcolato con versione precedente”.

---

#### AC-020 — Ricalcolo con nuova versione

Given uno scenario usa pack obsoleto,
when l’utente sceglie “Recalculate with latest pack”,
then il sistema ricalcola usando la nuova versione e aggiorna il risultato.

---

#### AC-021 — Duplicazione scenario con nuovo pack

Given uno scenario usa pack obsoleto,
when l’utente sceglie “Duplicate with latest pack”,
then il sistema crea nuovo scenario lasciando invariato quello storico.

---

#### AC-022 — Pack deprecated non default

Given un pack è marcato deprecated,
when l’utente prova a impostarlo come default,
then il sistema impedisce l’azione o richiede conferma forte.

---

#### AC-023 — Pack deprecated in nuovo scenario

Given l’utente seleziona un pack deprecated in un nuovo scenario,
when procede,
then il sistema mostra warning esplicito.

---

#### AC-024 — Eliminazione pack custom non usato

Given un pack custom non è usato da scenari,
when l’utente conferma eliminazione,
then il pack viene eliminato.

---

#### AC-025 — Eliminazione pack custom in uso

Given un pack custom è usato da scenari,
when l’utente tenta di eliminarlo,
then il sistema impedisce l’eliminazione.

---

#### AC-026 — Import pack valido

Given l’utente importa JSON con assumption pack valido,
when conferma import,
then il pack viene salvato come imported/custom.

---

#### AC-027 — Import pack non valido

Given l’utente importa JSON con pack non valido,
when il sistema valida i dati,
then mostra errori e non modifica i dati esistenti.

---

#### AC-028 — Export pack custom

Given uno scenario usa un pack custom,
when l’utente esporta JSON,
then il pack custom viene incluso nell’export.

---

#### AC-029 — Compare pack

Given l’utente seleziona due assumption pack,
when clicca Compare,
then il sistema mostra differenze su credit bands, model factors, driver factors e funding defaults.

---

#### AC-030 — Disclaimer visibile

Given l’utente visualizza un assumption pack,
when la UI mostra credit bands o factor,
then deve essere visibile che sono assunzioni di stima e non prezzi ufficiali vincolanti.

### 24. Priorità

Priorità funzionale: **Must have**

Gli assumption pack sono fondamentali perché permettono all’app di essere:

* trasparente;
* auditabile;
* modificabile;
* aggiornata nel tempo;
* onesta rispetto all’incertezza del pricing;
* adatta a scenari consulenziali;
* capace di confrontare stime storiche e nuove.

Senza assumption pack, il motore di calcolo avrebbe valori hardcoded e non spiegabili.

---

### 25. Note UX

La gestione assumption pack deve essere pensata per due tipi di utenti:

1. Utente business

   * deve poter scegliere un pack senza entrare nei dettagli;
   * deve vedere nome, versione e warning principali;
   * deve capire se il pack è ufficiale, custom o legacy.

2. Utente avanzato

   * deve poter modificare credit bands e factor;
   * deve vedere differenze tra pack;
   * deve poter duplicare e versionare.

Principi UX:

* mostrare sempre nome e versione del pack;
* mostrare badge System, Custom, Imported, Deprecated;
* rendere visibile la source date;
* mostrare warning se il pack è vecchio;
* impedire modifica diretta dei pack system;
* rendere semplice la duplicazione;
* usare editor tabellari chiari;
* evidenziare valori custom;
* mostrare impatto delle modifiche;
* mantenere disclaimer sempre visibile;
* evitare che l’utente interpreti i valori come listino ufficiale.

### 27. Definizione di completato

Il FP-008 può essere considerato completo quando:

* esiste sempre un assumption pack default;
* ogni scenario richiede un assumption pack;
* i pack di sistema sono visualizzabili ma non modificabili;
* i pack di sistema non sono eliminabili;
* l’utente può duplicare un pack di sistema;
* l’utente può creare pack custom;
* l’utente può modificare pack custom;

* ogni pack contiene driver factors;
* ogni pack contiene funding defaults;

* i pack custom sono inclusi in export JSON;
* i pack importati sono validati;
* è possibile confrontare due pack;
* il report mostra il pack usato;
* il pannello “How this was calculated” mostra i valori del pack;
* la UI chiarisce che le assunzioni non sono prezzi ufficiali;
* tutto funziona senza backend e resta compatibile con GitHub Pages.
